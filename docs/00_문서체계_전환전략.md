# 문서 체계 전환 전략

- 작성일: 2026-03-10
- 상태: 채택

## 1. 배경

현재 저장소는 `설계/`에 기획, 정책, API, 스키마, 확장 전략, ADR이 함께 들어 있다.  
장점은 정보량이 충분하다는 점이지만, 바이브 코딩이나 AI 구현 기준으로는 문서가 많아서 "지금 무엇을 기준으로 구현해야 하는가"가 다시 흔들릴 수 있다.

이번 전환의 목표는 두 가지다.

- 구현 직전에 보는 문서를 5개로 줄인다.
- 기존 상세 설계와 ADR은 버리지 않고 배경/근거 저장소로 유지한다.

## 2. 비교한 옵션

### 옵션 A. 현재 `설계/` 체계를 그대로 유지

장점:

- 이미 작성된 정책, 스키마, API, ADR 자산을 그대로 활용할 수 있다.
- 세부 설계 정보가 많아 깊은 판단이 필요할 때 참고 자료가 충분하다.

트레이드오프:

- 구현 시작점이 너무 넓다.
- 같은 내용을 여러 문서에서 다시 읽게 되어 AI가 불필요하게 되묻기 쉽다.
- MVP1 범위보다 확장 기능 문서가 더 눈에 띄면 스코프가 흔들린다.

### 옵션 B. 기존 문서를 버리고 5개 문서만 남긴다

장점:

- 가장 단순하다.
- 구현자가 읽어야 하는 양이 크게 줄어든다.

트레이드오프:

- 이미 정리된 정책/스키마/API/ADR 근거가 사라진다.
- 세부 계약이 부족해지면 결국 구현 단계에서 다시 질문이 생긴다.
- 백엔드와 프론트엔드가 실제로 필요한 상세 정보까지 같이 잃게 된다.

### 옵션 C. `/docs`에 얇은 실행 SSOT를 만들고 `설계/`는 상세/근거 저장소로 유지

장점:

- 구현 시작점이 명확하다.
- 문서는 얇게 유지하면서도, 필요한 깊이는 기존 설계와 ADR에서 바로 내려갈 수 있다.
- 범위, 유저 플로우, 용어, API/상태, 작업순서를 한 번에 고정할 수 있다.

트레이드오프:

- 문서 계층 규칙을 명확히 운영해야 한다.
- `/docs`와 `설계/`가 어긋나지 않도록 변경 규칙이 필요하다.

## 3. 최종 결정

옵션 C를 채택한다.

운영 구조:

- `/docs/01~05`: 현재 MVP1 구현용 얇은 SSOT
- `설계/adr/`: 기술 선택 근거 기록
- `설계/04`, `10`, `11`, `29`: 정책/스키마/API/프론트 상세 계약
- 나머지 `설계/`: 배경, 확장 전략, 참고 자료

## 5. 운영 규칙

- 구현 전에는 먼저 `/docs/01~05`만 읽는다.
- 새 기술 선택이 필요하면 코드보다 ADR을 먼저 만든다.
- 새 필드, 상태값, 에러 처리 변경이 생기면 `/docs/04_API_상태계약.md`를 먼저 바꾼다.
- 새 기능이 MVP1 범위를 넘으면 구현하지 말고 `/docs/05_구현백로그.md`로 보낸다.
- 상세 설계가 필요할 때만 `설계/`로 내려간다.
