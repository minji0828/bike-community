## 1. 고정 전제

- 웹 프론트엔드(`frontend/`)는 Next.js이므로 Vercel에 배포한다.
- 제품의 1차 클라이언트는 모바일 앱이지만, 현재 저장소에는 웹 구현체만 있으므로 웹은 검증/공유 진입용으로 계속 운영한다.
- 백엔드는 Spring Boot + PostgreSQL + PostGIS 조합이다.
- 백엔드는 장기적으로 EC2로 갈 수 있지만, 그 전에 더 간단한 배포 경로를 우선 고려한다.

---

## 2. 프론트엔드 전략

### 선택: Vercel

이유:

- 현재 웹 앱이 Next.js라서 가장 설정이 적다.
- 브랜치별 미리보기와 본 배포가 자연스럽다.
- 현재 `frontend/` 구조를 거의 그대로 사용할 수 있다.

운영 원칙:

- 프로덕션 도메인: `www` 또는 루트 도메인
- 백엔드 API 도메인: 별도 `api` 서브도메인
- 웹의 `/api/auth/me` 같은 same-origin route는 Vercel에서 계속 유지
- 웹 앱은 backend base URL만 환경변수로 바꿔 연결

근거:

- Vercel Next.js 문서: https://vercel.com/docs/frameworks/nextjs
- Vercel CLI deploy 문서: https://vercel.com/docs/cli/deploy

## 3. 백엔드 후보 비교

### 옵션 A. Render Web Service + Render Postgres

장점:

- Java/Spring Boot 앱을 Dockerfile 기준으로 바로 올릴 수 있다.
- Render Postgres가 공식적으로 PostGIS extension을 지원한다.
- Git 연결 후 push 기반 자동 배포가 단순하다.
- 추후 EC2로 옮겨도 Dockerfile, env var, 외부 Postgres 연결 구조를 유지하기 쉽다.

단점:

- 장기적으로는 플랫폼 종속 설정이 조금 생긴다.
- 정교한 네트워크/인프라 제어는 EC2보다 제한적이다.

이 프로젝트 적합도:

- 가장 높음

근거:

- Render Web Services: https://render.com/docs/web-services
- Docker on Render: https://render.com/docs/docker
- Render Postgres extensions: https://render.com/docs/postgresql-extensions

### 옵션 B. Railway Service + Railway PostGIS Template

장점:

- 초기 올리기는 빠르다.
- Dockerfile 경로 지정과 서비스 배포가 단순하다.
- PostGIS 템플릿이 존재한다.

단점:

- 기본 PostgreSQL 가이드는 extension을 직접 포함하지 않고, PostGIS는 템플릿 경로를 따로 골라야 한다.
- 팀이 실수로 기본 Postgres를 고르면 PostGIS 전제가 깨질 수 있다.
- 운영 제어/구성 명확성은 Render보다 약간 덜 직관적일 수 있다.

이 프로젝트 적합도:

- 중간 이상

근거:

- Railway Dockerfile docs: https://docs.railway.com/deploy/dockerfiles
- Railway PostgreSQL docs: https://docs.railway.com/guides/postgresql
- Railway PostGIS template: https://railway.com/deploy/postgis

### 옵션 C. Fly.io App + Fly Managed Postgres

장점:

- Docker/컨테이너 제어가 강하다.
- Managed Postgres에서 PostGIS를 켤 수 있다.
- 장기적으로 세밀한 운영 제어에 유리하다.

단점:

- 초기 러닝커브가 더 있다.
- "EC2 전에 빨리 올리기" 목적만 보면 Render보다 단순하지는 않다.

이 프로젝트 적합도:

- 중간

근거:

- Fly Managed Postgres extensions: https://fly.io/docs/mpg/extensions/

## 4. 추천안

### 추천 조합

- 프론트: Vercel
- 백엔드: Render Web Service
- DB: Render Postgres + PostGIS

추천 이유:

- 현재 웹 프론트는 Vercel이 가장 자연스럽다.
- 백엔드는 Spring Boot + Docker + PostGIS 조합이라 Render가 가장 마찰이 적다.
- Railway도 가능하지만 PostGIS 선택 실수가 생기기 쉽고, 이 프로젝트는 공간 데이터가 핵심이라 DB 확장 지원이 더 명확한 쪽이 낫다.
- Fly.io는 나쁘지 않지만 "지금 빨리 검증" 목적에는 운영 복잡도가 더 높다.

## 5. 단계별 운영 로드맵

### Step 1. 지금 당장

- 웹 프론트 `frontend/`를 Vercel에 배포
- 백엔드 Docker 이미지를 Render Web Service에 배포
- Render Postgres에 `postgis` extension 활성화
- 도메인 분리
  - `www.example.com` -> Vercel
  - `api.example.com` -> Render

### Step 2. 모바일 착수 후

- 모바일 앱은 `api.example.com`을 직접 사용
- 카카오 redirectUri와 딥링크를 모바일 기준으로 추가
- 웹은 공유/공개 열람용으로 유지

### Step 3. EC2 전환 시

- Dockerfile, env var, DB 마이그레이션 기준 유지
- backend만 EC2 또는 ECS 계열로 이동
- 프론트는 계속 Vercel 유지 가능
- DB는 관리형 유지 또는 별도 이전 판단
