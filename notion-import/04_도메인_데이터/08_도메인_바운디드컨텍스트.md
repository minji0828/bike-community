## 1. 유비쿼터스 언어(용어 정리)

- 코스(Course): 공유 가능한 경로 콘텐츠. 핵심 자산.
- 라이딩(Riding): 한 번의 실제 주행 기록. 원본 데이터.
- POI: Point of Interest. MVP는 공공화장실 중심.
- 피처드 코스(Featured): 운영이 기본 제공으로 노출하는 코스.
- 메타데이터(Metadata): 코스 추천/요약을 위한 구조화된 속성(core/derived/annotation/ops).
- 경고(Warning): 위험/불편/주의 정보(구조화).
- 하이라이트(Highlight): 볼거리/정차 추천 포인트(구조화).
- 코스 따라가기(Guidance L2): Polyline + 진행률 + 이탈 감지.
- 추천(Recommendation): 코스 후보를 필터/랭킹해 3~5개 제시.

## 2. 바운디드 컨텍스트 목록

> 현재 코드베이스에 이미 존재하는 컨텍스트는 ✅로 표시.

### 2.1 Identity(사용자/디바이스) ✅

- 책임: 사용자 식별(userId), 비회원 식별(deviceUuid), (추후) 인증
- 소유 데이터: User, Device
- 외부 노출: 사용자 등록/조회

### 2.2 Tracking(위치 업데이트/이력) ✅

- 책임: 현재 위치/이력 저장, retention 정책 적용
- 소유 데이터: UserLocation
- 외부 노출: 위치 업데이트/조회 API

### 2.3 Riding(라이딩 기록) ✅

- 책임: 주행 경로(LineString) 저장, 기본 메타 저장(거리/시간/평균속도)
- 소유 데이터: Riding
- 외부 노출: 라이딩 생성 API

### 2.4 POI(수집/탐색) ✅

- 책임: 공공데이터 동기화(전체/증분), 주변/경로 주변 조회
- 소유 데이터: Poi
- 외부 연동: 서울 열린데이터 API 등

### 2.5 Course Catalog(코스 콘텐츠) (신규)

- 책임:
  - 코스 생성(UGC/큐레이션)
  - 코스 공유/공개범위
  - 코스 메타데이터 v1 저장/산출
  - 태그/경고/하이라이트
  - 피처드 운영
- 소유 데이터: Course, Tag, CourseTag, CourseWarning, CourseHighlight
- 외부 의존: POI(amenitiesSummary 산출), Riding(코스 생성 원본)

### 2.6 Recommendation(추천) (신규)

- 책임:
  - (MVP 이후) 필터 기반 추천: 조건 -> 후보 -> 랭킹
  - (2차) 자연어 추천 에이전트: 자연어 -> 조건(JSON) 정규화
- 소유 데이터: 없음(대부분 조회/랭킹). 추후 선호도/세션 로그가 생기면 별도 테이블.
- 외부 의존: Course Catalog(코스 후보), POI(편의성 지표)

### 2.7 Guidance(코스 따라가기) (클라이언트 중심)

- 책임: 진행률/이탈 감지, 지도 렌더링 UX
- 소유 데이터: (MVP) 서버 저장 없음. (vNext) follow session 로그 수집은 가능.
- 메모: L2는 클라이언트 계산이 효율적(지연/비용/프라이버시 측면).

### 2.8 Backoffice(운영 UI/관리 API)

- 책임: POI 동기화 트리거/상태, 코스 큐레이션/태그/경고/피처드, 재계산
- 소유 데이터: 없음(각 컨텍스트의 관리용 API)

## 4. 애그리게이트(상태/불변식 경계)

### Course (Aggregate Root)

- 불변식(초안)
  - path(LineString)는 비어있을 수 없다(최소 좌표 2개)
  - shareId는 중복될 수 없다
  - visibility는 enum 범위 내
  - sourceType/verifiedStatus 일관성(예: curated 코스는 verifiedStatus=curated)

### Poi

- externalId로 증분 동기화

### Riding

- deviceUuid는 필수(비회원 식별)

## 6. 설계 원칙(왜 이렇게 나눴나)

- 코스(Course)는 핵심 자산이라 POI/라이딩/추천과 분리된 "Catalog"로 취급
- POI는 외부 동기화/약관/쿼터 이슈가 있어 Catalog에서 직접 외부 호출하지 않음(동기화 후 서빙)
- Guidance는 L2 기준 클라이언트 계산이 최적(서버는 코스/POI 데이터를 제공)
- 추천은 초기에 규칙 기반으로도 충분하며, AI는 입력 UX로만 도입(결정 로직은 결정적)
