목적:

핵심 원칙:

- 외부 API(POI/지도/길찾기)는 런타임 의존을 줄이고 동기화/캐시로 방어한다.

## 1. 지도(Map) SDK 선택(React Native)

MVP에서 필요한 지도 기능(최소):

- Polyline 렌더링(코스)
- Marker 렌더링(POI)
- 카메라 제어(재센터링)

### 옵션 비교

옵션 A: `react-native-maps`(플랫폼 기본 지도/프로바이더)

- 장점: 구현 난이도 낮음, 생태계/예제 풍부, 폴리라인/마커 중심 MVP에 충분
- 단점: 고급 스타일링/오프라인/내비 기능은 한계가 있을 수 있음

옵션 B: Mapbox 계열(Mapbox/MapLibre)

- 장점: 스타일링/성능/확장성 좋음, 향후 내비/오프라인과의 결합 가능
- 단점: 라이선스/요금/키 관리, SDK 러닝커브

옵션 C: 국내 지도 SDK(네이버/카카오) + RN 래퍼

- 장점: 한국 지역 품질/POI 생태계 강점 가능
- 단점: RN에서의 유지보수/호환성/권한/빌드 이슈 리스크

MVP 기본 선택(권장):

- 옵션 A로 시작하되, 지도 계층을 "추상화"해서 교체 비용을 낮춘다.
  - 예: `MapAdapter` 인터페이스(Polyline/Marker/Camera만)

결정 이유:

- L2 가이드는 지도에서 선/점/카메라만 필요해서, 특정 벤더의 고급 기능에 락인될 이유가 작다.
- 추천/POI/코스 메타데이터가 먼저 가치 검증 포인트다.

## 2. 턴바이턴(레벨 4) / 라우팅 기술

MVP 제외(현재):

- 턴바이턴, 음성 안내, 재탐색(리라우팅)

vNext에서 선택지:

옵션 A: 외부 내비 앱 딥링크/공유(가장 가벼움)

- 장점: 구현/운영 비용 최소, 약관/쿼터 부담 상대적으로 낮음
- 단점: 앱 경험 일관성 낮음(앱 전환), 코스/POI 통합 경험이 약해질 수 있음

옵션 B: 지도/길찾기 API(호스티드)로 경로 + maneuver 생성

- 장점: 내장 내비 UX 가능, maneuver 기반 안내 구현 가능
- 단점: 요청당 비용/쿼터, 약관 준수 필요, 품질/커버리지 차이

옵션 C: 오픈소스 라우팅 엔진 자가 호스팅(OSRM/Valhalla/GraphHopper 등)

- 장점: 쿼터/단가 통제 가능, 커스터마이즈 가능
- 단점: 데이터(도로망) 업데이트/운영 난이도 높음, 1인 MVP에 과함

권장 로드맵:

- vNext-1: 옵션 A(외부 내비)로 검증
- vNext-2: 필요성이 증명되면 옵션 B로 내장 내비 도입
- 옵션 C는 트래픽/단가/커스터마이즈 요구가 커졌을 때 검토

## 3. POI 데이터 소스(화장실/카페)

### 3.1 화장실(우선)

현황:

- 현재 코드베이스는 서울 공공 데이터 기반 동기화 + PostGIS 조회를 이미 구현했다.

확장 전략:

- 전국 단위로는 "지자체별" 데이터 형식이 다를 수 있으므로,
  - ETL 계층(수집/정규화/적재)을 POI 컨텍스트 내부로 분리
  - `external_id` + `source`(데이터 출처) 필드를 표준화

### 3.2 카페/편의(후순위)

옵션 A: 공개 데이터(지자체/공공데이터포털)

- 장점: 비용/약관 리스크가 낮을 수 있음
- 단점: 커버리지/갱신 주기가 불안정할 수 있음

옵션 B: OSM 기반(amenity=cafe 등)

- 장점: 무료/글로벌 확장 가능
- 단점: 지역별 품질 편차(특히 최신성)

옵션 C: 상용 로컬 검색 API

- 장점: 품질/최신성 강점 가능
- 단점: 쿼터/요금/약관 리스크, 저장/캐시 가능 범위 제한 가능

MVP 기본 선택(권장):

- 카페는 MVP Must에서 제외하고, 대신
  - "하이라이트(highlights)"로 사용자가 직접 추천 포인트를 남기게 해서 코스 콘텐츠를 강화
- 카페 API 연동은 약관/비용/품질 검토 후 vNext로 진행

## 4. 공간 연산/검색(PostGIS vs App)

옵션 A: PostGIS 중심(ST_DWithin, geography distance)

- 장점: 공간 인덱스로 빠름, 서버 스케일링에 유리
- 단점: 쿼리 튜닝/인덱스 설계 필요

옵션 B: 애플리케이션(JTS)에서 거리/버퍼 계산

- 장점: DB 의존 감소
- 단점: 대량 데이터에서 성능/비용 불리, 구현 오류 위험

MVP 기본 선택(권장):

- 옵션 A(PostGIS) 중심
- App(JTS)은 "변환/검증" 보조로 제한

## 5. 캐시 전략(Redis vs In-memory)

캐시 대상:

- along-route POI 결과
- featured 코스 리스트
- 코스 상세 요약(amenitiesSummary)

옵션 A: In-memory(Caffeine)

- 장점: 설정 간단, 비용 낮음, 단일 인스턴스 MVP에 적합
- 단점: 멀티 인스턴스에서 캐시 공유 불가

옵션 B: Redis

- 장점: 멀티 인스턴스 캐시 공유, TTL/정책 다양
- 단점: 운영 요소 증가(배포/모니터링)

MVP 기본 선택(권장):

- 단일 인스턴스: 옵션 A
- 트래픽 증가/수평 확장 시: 옵션 B로 전환

## 6. 비동기/배치(워커)

옵션 A: Spring `@Scheduled` + (선택) `@Async`

- 장점: MVP에 가장 단순
- 단점: 작업 실패/재시도/관측이 빈약해질 수 있음

옵션 B: Quartz

- 장점: 작업/스케줄 관리 기능이 풍부
- 단점: 도입 비용

MVP 기본 선택(권장):

- 옵션 A로 시작
- 실패/재시도/대시보드가 필요해지면 Quartz 또는 외부 워커로 확장

## 7. AI 추천(자연어 에이전트) 모델/구조

목표:

- 사용자가 자연어로 조건을 말하면(예: "60분, 평지, 화장실 걱정 없는 강변")
  - 조건(JSON)으로 정규화
  - 결정적 추천 엔진(필터/점수화)에 전달
  - 추천 이유를 설명

권장 구조:

- LLM은 "조건 추출"과 "설명 생성"만 담당
- 추천/랭킹은 서버 규칙 기반으로 결정(재현성/디버깅/비용)

모델 옵션:

- 옵션 A: 클라우드 LLM(OpenAI/Anthropic/Google 등)
  - 장점: 품질/안정성, 함수호출(JSON) 기능
  - 단점: 비용/지연/키 관리
- 옵션 B: 오픈소스 모델(자가 호스팅)
  - 장점: 단가/데이터 통제
  - 단점: 운영/튜닝 비용 큼(1인 MVP에 부담)

MVP 기본 선택(권장):

- 옵션 A(경량 클라우드 LLM) + 요청량 제한 + 로그/모니터링
- 모델은 추상화 인터페이스로 감싸서 교체 가능하게 설계
