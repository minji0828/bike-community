## 1. 마일스톤 정의

### M0. 서버 베이스라인 확정(현재 기능이 "돌아가는" 상태)

목표:

- POI sync / nearby / along-route / riding 저장이 로컬에서 재현 가능

완료 기준(DoD):

- Swagger UI에서 POI 조회가 정상 응답
- DB에 pois/ridings 테이블이 생성되고 데이터가 적재됨
- `./gradlew test` 통과

### M1. Course Catalog(코스 콘텐츠) 스키마/도메인 구현

목표:

- 코스를 저장/조회/공유할 수 있는 서버 기능을 만든다.
- 코스 메타데이터 v1(distance/bbox/amenitiesSummary) 파이프라인을 만든다.

작업:

- DB: `courses`, `tags`, `course_tags`, `course_warnings`(필수)
- API: `POST /api/v1/courses`, `GET /api/v1/courses/{id}`
- API: `POST /api/v1/courses/gpx`, `GET /api/v1/courses/{id}/gpx`
- API: `POST /api/v1/courses/{id}/share`, `GET /api/v1/courses/public/{shareId}`
- 메타데이터:
  - 동기: distance/bbox/loop
  - 비동기: toilet_count(코스 주변 버퍼)

완료 기준(DoD):

- 코스 1개 생성 -> 상세 조회에서 distance/bbox가 반환
- GPX 업로드 생성 -> GPX 다운로드 조회가 가능
- share 발급 -> public 조회 가능(visibility 정책 준수)
- 메타데이터 비동기 계산 실패 시에도 코스 조회는 가능

### M2. Featured(기본 제공 코스) 운영

목표:

- 앱이 비어있지 않도록 기본 코스를 제공(콜드스타트 해결)

작업:

- `GET /api/v1/courses/featured`
- featured_rank/verified_status 운영 필드 반영
- (선택) admin API로 featured 설정

완료 기준(DoD):

- featured 코스 리스트가 정렬된 카드 요약으로 응답
- 최소 10개 코스를 시드로 넣고(수동/스크립트/관리 API), 앱에서 확인 가능

### M3. RN 클라이언트 최소 기능(콜드스타트 -> 따라가기)

목표:

- 피처드 코스 선택 -> 코스 상세 -> 코스 따라가기(L2)

작업:

- 앱 첫 화면: 홈(대시보드) = Featured List(추천 코스) + 라이딩 시작 CTA
- 화면: Featured List
- 화면: Course Detail(Polyline + 요약 + 화장실 요약)
- 화면: Follow(진행률/이탈)

완료 기준(DoD):

- 코스 선/POI 마커 렌더링
- 진행률 표시(%) 또는 남은 거리
- 이탈/복귀 UX가 깜빡이지 않음(히스테리시스)

### M4. UGC 코스 생성(라이딩 -> 코스)

목표:

- 사용자가 탄 경로를 저장하고 코스로 만들 수 있음

작업:

- 라이딩 저장 후 코스 생성 API(`from-riding`)
- 코스 생성 시 tags/notes/warnings 최소 입력
- 라이딩 GPS 추적/기록 정책 문서 고정: `설계/23_라이딩_GPS_추적_기록_설계.md`

완료 기준(DoD):

- 라이딩 저장 -> 코스 생성 -> 공유 -> 다른 기기에서 열람(또는 동일 기기)

### M5. Backoffice 최소 API

목표:

- 운영이 코스/태그/경고/재계산을 제어할 수 있음

작업:

- admin endpoints(인증은 MVP에서 제한적으로):
  - tag 사전 CRUD
  - course warnings 관리
  - metadata 재계산 트리거

완료 기준(DoD):

- featured 코스의 경고/태그/노출순서를 수정 가능

### M6. 인증 + 코스 댓글 MVP2

목표:

- 카카오 소셜 로그인(OIDC)으로 write를 가능하게 한다.
- 코스 상세에 댓글이 쌓여 "혼자" 느낌을 줄인다(익명 표시).

작업:

- Auth: 카카오 OIDC(code+PKCE) 교환 + id_token 검증 + 서비스 JWT 발급
- DB: `course_comments` (+ 선택: `reports`)
- API: 댓글 작성/조회/삭제, 댓글 신고
- 정책: 익명 표시(작성자 정보 비노출), rate limit, 숨김 처리(운영)

완료 기준(DoD):

- 로그인 성공 후 댓글 작성 가능
- 댓글은 UI/API에서 작성자 표시가 항상 `익명`
- 작성자만 삭제 가능(soft delete)
- 신고된 댓글은 운영이 숨김 처리 가능

### M7. 코스 모임 + 임시 채팅 MVP3

목표:

- 코스 기반 모임 생성/참여로 "사람을 모으는" 경험을 만든다.
- 모임 내 임시 채팅을 제공한다.

작업:

- DB: `course_meetups`, `course_meetup_participants`
- API: 모임 생성/리스트/상세/join/leave
  - `POST /api/v1/courses/{courseId}/meetups`
  - `GET /api/v1/courses/{courseId}/meetups?status=open`
  - `GET /api/v1/meetups/{meetupId}`
  - `POST /api/v1/meetups/{meetupId}/join`
  - `POST /api/v1/meetups/{meetupId}/leave`
- 동시성: join 정원 처리(row lock) + 중복 참가 방지(unique)
- 통신: 채팅은 WebSocket(STOMP)

## 2. 리스크 기반 우선순위(왜 이 순서인가)

- 코스(콘텐츠)와 메타데이터 파이프라인이 먼저 있어야 추천/AI도 의미가 생김
- L2 가이드는 클라이언트 구현이지만, 서버에서 코스 데이터를 안정적으로 서빙해야 함
- 카페/상용 POI API는 약관/비용 리스크가 있어 후순위
