# 설계 문서 정합성 / 개발 준비도 검토

- 문서 ID: DESIGN-REVIEW-2026-03-10
- 버전: v1.0
- 작성일: 2026-03-10
- 상태: 검토 완료

## 1. 결론 요약

현재 `설계/` 문서는 "기획부터 확장 전략까지 넓게 잡혀 있는 상태"다.  
문서 양은 충분하지만, 실제 개발 관점에서는 "어느 문서가 최종 계약인지"가 약해서 AI나 신규 개발자가 바로 들어가면 중복 문서를 동시에 참조할 가능성이 높다.

판정:

- 제품 기획/문제 정의: 좋음
- MVP 범위/정책: 대체로 좋음
- 백엔드 구현 계약: 개발 가능 수준에 가까움
- 프론트엔드 구현 계약: 현재 저장소 기준으로는 미정 요소가 남아 있음
- AI 기능 구현 계약: 아직 전략 단계

## 2. 잘 잡혀 있는 부분

- 문제 정의 -> MVP 범위 -> FR/NFR -> 정책 -> 스키마 -> API로 이어지는 기본 체인이 있다.
- 코스, 하이라이트, 컬렉션, 인증, 댓글, 모임 등 확장 방향이 기능 묶음 단위로 정리돼 있다.
- 정책 문서에 공개 범위, 프라이버시, 오프루트, 태그, 하이라이트, 댓글 모더레이션 같은 제품 규칙이 숫자/enum 중심으로 제시돼 있다.
- 백엔드 핵심 도메인(Course/Auth/Collection/Highlight/Comment/Meetup)과 문서 주제가 대체로 맞물린다.

## 3. 주요 리스크

### 3.1 SSOT가 약했다

- `bikeoasis_prd_v_1.md`, `02_MVP_요구사항정의서.md`, `03_기능_비기능_요구사항명세서.md` 사이에 범위/기능 서술이 중복된다.
- `16_커뮤니티_동시성_통신.md`는 이후 `17~22` 문서와 역할이 겹친다.
- `24_운영전환_핵심기능_배포기준.md`는 `14`, `25`, `26`, `27`과 일부 목적이 겹친다.

영향:

- AI가 참조 문서를 잘못 잡으면 오래된 설명을 구현 계약으로 오인할 수 있다.

### 3.2 프론트엔드 타깃이 문서와 저장소에서 다르다

- 문서 다수는 React Native / Expo / RN을 전제로 쓴다.
- 실제 저장소의 프론트엔드는 `frontend/` 아래 Next.js 웹 앱이다.

영향:

- 인증, 공유 링크, 딥링크, 지도 SDK, 위치 권한, 백그라운드 GPS, 오프라인 정책을 그대로 구현 기준으로 쓰면 수정 비용이 커진다.

### 3.3 API / 스키마 문서가 아직 "초안 서술" 중심이다

핵심 엔드포인트와 테이블은 정리되어 있지만, 다음이 문서에 더 명확히 들어가야 한다.

- 필수/선택/null 허용 여부
- enum 실제 값과 casing
- 인증 필요 여부
- request validation 규칙
- 에러 코드와 실패 케이스
- 페이지네이션 / 정렬 / 필터 조건
- 상태 전이 규칙

영향:

- AI가 DTO, validation, 테스트 코드를 생성할 때 추정이 많아진다.

### 3.4 AI 기능 문서는 아직 구현 계약이 아니다

`설계/13_기술선정_비교.md`와 일부 전략 문서에 추천/에이전트 방향은 있다.  
하지만 아래 항목은 아직 없다.

- 추천 입력 피처 스키마
- 모델이 반환해야 하는 출력 포맷
- 룰 기반 fallback
- 평가 지표와 샘플 데이터셋
- 환각/잘못된 추천 방지 정책

영향:

- AI 관련 구현은 바로 시작하기 어렵고, 시제품 수준 이상으로 가면 재작업 가능성이 크다.

## 4. 현재 개발 준비도 판정

### 4.1 백엔드

- 판정: 거의 준비됨
- 조건: `04`, `10`, `11`, `21`, `23`, `14`를 우선 계약 문서로 삼으면 바로 개발 가능
- 잔여 과제: API/스키마 문서의 validation, enum, 상태 전이 명시

### 4.2 프론트엔드

- 판정: 부분 준비
- 현재 저장소 기준 블로커:
  - 웹인지 모바일인지 최종 타깃이 문서에서 고정되지 않음
  - RN 전용 가정과 웹 라우팅/인증 플로우가 섞여 있음
- 권장: 현재 저장소 기준으로는 웹 계약을 먼저 고정하고, 모바일은 별도 문서 트랙으로 분리

### 4.3 AI / 추천

- 판정: 아직 준비 안 됨
- 권장: 전략 문서와 별도로 "입력/출력 계약 + 평가 방법 + fallback"을 명시한 구현 문서를 만든 뒤 시작

## 6. 앞으로 꼭 보강할 것

### A. 프론트엔드 플랫폼 계약

최소 포함 항목:

- 현재 릴리즈 타깃: 웹 / 모바일 / 둘 다
- 로그인 진입점과 콜백 처리
- 공유 링크가 웹에서 열리는지, 앱 딥링크로만 열리는지
- 지도 SDK / 위치 권한 / 백그라운드 정책
- 오프라인 / 캐시 / 저장 정책

### B. API 구현 계약 보강

각 endpoint마다 아래를 고정해야 한다.

- auth required 여부
- request 예시와 validation
- response 필드 nullability
- enum 허용값
- 대표 오류 케이스
- 부작용(shareId 발급, metadata 재계산 등)

### C. 상태 머신 / 정책 표

특히 아래는 표로 고정하는 것이 좋다.

- course visibility / verified status
- collection visibility
- comment status
- highlight status
- meetup status

### D. AI 계약 문서

최소 포함 항목:

- 입력 피처 목록
- 추천 결과 포맷
- 룰 기반 fallback
- 프롬프트/모델 사용 범위
- 평가 셋과 오프라인 지표

## 7. 실무적으로 어떻게 쓰면 되나

- 코어 MVP 구현은 `04 -> 10 -> 11 -> 14` 순서로 본다.
- 인증은 `21`, GPS 기록은 `23`을 별도 계약 문서로 본다.
- 확장 기능은 `17~27`에서 필요한 트랙만 꺼내 쓴다.
- 참조용 문서는 맥락 확인용이지, 최종 코드 계약으로 사용하지 않는다.
