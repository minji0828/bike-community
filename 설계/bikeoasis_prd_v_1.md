# PRD — 가자(BikeOasis) MVP

- 문서 버전: v1.0 draft
- 작성일: 2026-03-06
- 문서 목적: 분산된 설계 문서를 제품 요구사항 문서(PRD)로 재구성하여, 제품/디자인/개발이 같은 기준으로 MVP를 실행할 수 있게 한다.

---

## 1. 제품 개요

### 1.1 한 줄 정의
가벼운 라이딩부터 자전거 여행까지의 사용자를 위해, **코스를 쉽게 따라가고**, **경로 주변의 필수 편의시설(우선: 공공화장실)** 을 확인하며, **내가 탄 길을 코스로 저장·공유하고 함께 갈 사람과 연결**할 수 있게 돕는 지도 기반 앱.

### 1.2 제품 비전
기존 지도 앱의 “현재 위치 중심 검색” 한계를 넘어, 사용자가 **실제로 가는 경로 기준으로 필요한 정보**를 제공하고, 라이딩 기록을 **재사용 가능한 코스 콘텐츠**로 전환해 **산책형 라이딩부터 여행형 코스 탐방까지** 확장 가능한 경험을 만든다.

### 1.2-1. 도메인 방향 전환 배경
- 최초 아이디어는 "가볍게 산책하듯 타는 사용자"의 불편을 푸는 데서 시작했다.
- 하지만 자전거는 도보보다 이동 반경이 넓고, 실제 사용자는 단순 산책보다 **어디를 갈지/어떤 코스를 탈지/누구와 함께 갈지**를 미리 계획하는 경우가 많다.
- 그래서 제품 정의도 "산책형 보조 앱"에 머무르기보다, **코스 탐색·따라가기·기록·공유·동행**이 자연스럽게 연결되는 **자전거 여행/라이딩 코스 플랫폼** 쪽이 더 적합하다고 판단한다.

### 1.3 MVP에서 검증할 핵심 가치
1. 사용자는 경로 주변 필수 POI 정보가 있으면 더 안심하고 라이딩할 수 있는가?
2. 사용자는 자신이 탄 길을 코스로 저장하고 다시 열어보거나 공유하는가?
3. 코스 따라가기(L2) 경험이 실제로 유용한가?
4. 여행/탐방형 코스에서도 공유/모임/채팅이 실제 사용 가치를 가지는가?

## 2. 문제 정의

### 2.1 해결하려는 문제
1. 라이딩 중 필요한 시설(특히 화장실)을 **내가 가는 길 기준**으로 찾기 어렵다.
2. 가볍게 타는 사용자에게 코스 저장/재사용/공유가 번거롭다.
3. 여행/탐방형 라이딩에서는 “검증된 코스가 있는지”, “경로 기준 편의시설이 있는지”, “누구와 함께 갈지”를 한 번에 해결하기 어렵다.
4. 코스/지도 조회와 경로 주변 탐색은 성장 시 read-heavy + geo query 병목이 생기기 쉽다.

### 2.2 왜 지금 중요한가
- 초보/라이트 유저는 정교한 루트 플래너보다 “그냥 켜서 바로 쓰는 경험”을 선호한다.
- 라이딩 중 편의시설 불안은 실제 이탈 요인이 된다.
- 좋은 길이 기록으로만 남지 않고 “코스 콘텐츠”로 축적되어야 서비스 자산이 생긴다.
- 여행/관광형 GPX 코스와 지역 대표 코스를 활용하면 초기부터 “빈 앱”처럼 보이지 않는 콘텐츠 경험을 만들 수 있다.

## 3. 목표와 비목표

### 3.1 제품 목표
- **G1.** 현재 위치 기준 주변 POI와 경로 주변 POI를 모두 빠르게 보여준다.
- **G2.** 라이딩 기록을 저장하고 코스로 전환해 공유/재사용 가능하게 만든다.
- **G3.** 코스 따라가기 가이드(레벨 2)를 제공해 실제 사용성을 검증한다.
- **G4.** 초기 콜드스타트를 해결하기 위해 기본 제공(Featured) 코스를 제공한다.
- **G5.** 향후 성장 시에도 견딜 수 있도록 조회/탐색 구조를 캐시/프리컴퓨트 친화적으로 시작한다.
- **G6.** 산책형 라이딩과 여행형 코스를 모두 수용할 수 있는 상위 제품 구조를 만든다.

### 3.2 MVP 비목표
- 댓글/피드/팔로우 등 커뮤니티 풀세트
- ML 기반 개인화 추천
- 자연어로 새 코스를 생성하는 AI 라우팅
- 오프라인 지도 다운로드
- 턴바이턴(TBT) 내비게이션
- 실시간 다자간 위치 공유
- 결제/구독

## 4. 타겟 사용자

### P1. 주말 산책 라이더
- 목적지는 대략적이거나 없음
- 복잡한 설정 없이 바로 쓰고 싶음
- 화장실/편의시설 불안이 큼

### P2. 코스 수집가 / 공유자
- 내가 탄 좋은 길을 다시 타고 싶음
- 친구에게 링크로 공유하고 싶음
- 코스를 콘텐츠처럼 모으는 성향이 있음

### P3. 여행/탐방형 라이더
- 검증된 대표 코스나 GPX 기반 여행 코스를 따라가고 싶음
- 여행 중 필요한 편의시설/휴식 포인트를 경로 기준으로 보고 싶음
- 혼자보다는 모임/채팅을 통해 함께 가는 경험에 반응할 가능성이 높음

### P4. 운영/큐레이터
- 비어 있는 앱처럼 보이지 않게 Featured 코스를 관리해야 함
- 태그, 경고, 노출 순서를 운영해야 함

## 5. 핵심 사용자 시나리오

### 시나리오 A. Featured 코스 탐색 → 코스 상세 → 따라가기
1. 사용자가 앱을 실행한다.
2. 홈에서 Featured 코스를 본다.
3. 코스 상세에서 경로, 거리, 예상 시간, 경로 주변 화장실 요약을 확인한다.
4. “따라가기 시작”을 누른다.
5. 앱은 코스 Polyline, 내 위치, 진행률, 남은 거리, 이탈 여부를 보여준다.

### 시나리오 A-2. 여행형 대표 코스 탐색 → 공유/모임
1. 사용자가 여행형 Featured 코스를 연다.
2. 코스 상세에서 거리, 예상 소요 시간, 대표 편의시설, 주의사항을 확인한다.
3. 마음에 들면 링크를 공유하거나 모임에 참여한다.
4. 필요한 경우 코스모임 채팅에서 일정/집결 정보를 조율한다.

### 시나리오 B. 앱 실행 → 내 주변 POI 확인
1. 사용자가 앱을 실행한다.
2. 위치 권한을 허용한다.
3. 지도에서 현재 위치와 주변 화장실을 본다.

### 시나리오 C. 라이딩 기록 → 코스 생성 → 공유
1. 사용자가 라이딩을 시작한다.
2. 앱이 경로 좌표를 수집한다.
3. 사용자가 라이딩을 종료한다.
4. 서버에 라이딩을 저장한다.
5. 저장된 라이딩으로 코스를 생성한다.
6. 공유 링크를 발급해 다른 사람에게 전달한다.

### 시나리오 D. 공유 링크로 코스 열람
1. 사용자가 공유 링크를 연다.
2. 공개 범위 정책에 맞는 코스 상세를 본다.
3. 필요하면 그대로 따라가기를 시작한다.

## 6. MVP 범위

### 6.1 Must Have

#### A. 지도/POI
- 현재 위치 표시
- 현재 위치 기준 반경 POI 조회/표시
- 경로(along-route) 기준 POI 조회/표시
- 공공화장실 데이터 동기화(전체/증분)

#### B. 라이딩/코스
- 라이딩 시작/일시정지/종료
- 라이딩 경로 저장
- 코스 생성(최소 title 필수)
- 코스 목록/상세 조회
- 코스 공유 링크 발급 및 공개 코스 조회
- Featured 코스 리스트/상세 제공
- 코스 메타데이터 v1 산출 및 저장
  - 거리
  - bbox
  - loop 여부
  - toiletCount 등 POI 요약

> MVP 기준의 코스 범주는 “가벼운 산책형 코스”에 한정하지 않는다.
> 초기 seed/featured는 여행/관광형 GPX 코스를 포함할 수 있다.

#### C. 따라가기(Guidance L2)
- 코스 Polyline 렌더링
- 내 위치 표시
- 진행률 계산
- 남은 거리 표시
- off-route 감지
- 복귀 감지
- 재센터링/간단 안내 UI

#### D. 운영/백오피스 최소
- POI 전체/증분 동기화 실행
- 마지막 성공/실패 상태 확인
- Featured 코스 운영 필드 관리
- 태그/경고 최소 운영

### 6.2 Should Have (가능하면 포함)
- 조회수 같은 최소 지표 집계
- GPX 업로드/다운로드
- 내 코스/내 라이딩 목록

### 6.3 Later
- 댓글/후기
- 소셜 로그인
- 추천 고도화
- 자연어 추천 에이전트
- 턴바이턴 내비게이션

### 6.4 실험 기능 트랙 (MVP2 후보)
- 카카오 로그인
- 코스모임 생성/참가
- 코스모임 단체채팅

> 위 항목은 2026-03-06 기준 코어 MVP(지도/POI/코스/따라가기)와 별도 트랙으로 관리한다.
> 목적은 커뮤니티 기능 자체를 출시 범위에 강제 포함하는 것이 아니라,
> “같이 탈 사람을 모으고 실시간으로 소통하려는 수요가 실제로 있는지”, 특히 여행/탐방형 코스에서 그 가치가 더 강한지를 빠르게 검증하는 것이다.
> 따라서 코어 MVP 성공지표와 분리된 보조 실험 지표(예: 모임 생성 수, 모임 참가율, 채팅 활성 세션 수)를 둔다.

## 7. 기능 요구사항

### FR-1 현재 위치 및 권한
- 앱은 현재 위치를 지도에 표시해야 한다.
- 위치 권한이 없으면 요청하거나 제한 모드로 전환해야 한다.
- GPS 정확도 상태를 사용자에게 표시해야 한다.

### FR-2 주변 POI 조회
- 앱은 현재 위치 기준 반경 내 POI를 조회할 수 있어야 한다.
- 기본 반경은 500m로 시작한다.
- 카테고리 필터 구조를 가져야 한다.

### FR-3 경로 주변 POI 조회
- 앱은 코스/경로 좌표 리스트를 기준으로 along-route POI를 조회할 수 있어야 한다.
- 응답은 지도와 요약 UI에 사용 가능한 구조여야 한다.

### FR-4 라이딩 기록
- 사용자는 라이딩을 시작/일시정지/종료할 수 있어야 한다.
- 앱은 주기적으로 위치를 수집해 경로를 구성해야 한다.
- 렌더링 성능을 위해 샘플링/단순화 정책을 적용할 수 있어야 한다.

### FR-5 라이딩 저장
- 서버는 라이딩 메타 정보와 경로(LineString)를 저장해야 한다.
- 좌표가 2개 미만이면 저장 정책에 따라 거부 또는 임시 저장한다.
- 내부 좌표 순서는 항상 (lon, lat)여야 한다.

### FR-6 코스 생성 및 관리
- 사용자는 라이딩 또는 업로드한 경로로 코스를 생성할 수 있어야 한다.
- 코스는 title, description, tags, visibility, sourceType을 가질 수 있어야 한다.
- visibility는 public / unlisted / private 정책을 따라야 한다.

### FR-7 코스 조회
- 앱은 코스 목록과 상세를 조회할 수 있어야 한다.
- 상세에는 geometry/polyline, 거리, 예상 시간, POI 요약이 포함되어야 한다.

### FR-8 코스 공유
- 사용자는 공유 링크를 발급할 수 있어야 한다.
- 공개 범위에 맞는 경우에만 공유 코스를 열람할 수 있어야 한다.

### FR-9 Featured 코스
- 홈과 탐색 화면에서 Featured 코스를 노출해야 한다.
- 운영은 최소 10개 이상의 Featured 코스를 관리할 수 있어야 한다.

### FR-10 코스 따라가기
- 사용자는 코스 따라가기를 시작할 수 있어야 한다.
- 앱은 내 위치와 코스 선을 비교해 진행률을 계산해야 한다.
- 이탈 시 경고 배너를 표시해야 한다.
- 복귀 시 정상 상태로 복원해야 한다.

## 8. 비기능 요구사항

### NFR-1 성능
- 코스 상세와 주변/경로 주변 조회는 모바일에서 체감상 즉시 반응해야 한다.
- read-heavy 시나리오를 고려해 캐시/프리컴퓨트가 가능한 구조여야 한다.

### NFR-2 안정성
- 외부 POI API 장애가 사용자 조회 전체를 막아서는 안 된다.
- 동기화 실패 시 마지막 성공 데이터로 서빙해야 한다.

### NFR-3 프라이버시
- 위치 데이터는 민감 정보로 취급한다.
- 운영 로그에 원본 위경도를 과도하게 남기지 않는다.
- 위치 이력 보관 기간은 기본 30일로 둔다.

### NFR-4 확장성
- 지도 SDK, 캐시, AI 추천 방식은 추상화되어 교체 가능해야 한다.
- L2 가이드와 장기 L4 TBT 내비는 구조적으로 분리한다.

### NFR-5 일관성
- DB/PostGIS/JTS는 (lon, lat) 순서를 따른다.
- API JSON은 `{ lat, lon }` 필드로 표현한다.
- 응답 포맷은 `ApiResponse{ code, message, data }`를 따른다.

## 9. 정책

### 9.1 공개 범위 정책
- `public`: 누구나 조회 가능
- `unlisted`: 링크를 아는 사람만 조회 가능
- `private`: 소유자만 조회 가능

### 9.2 따라가기 정책
- 이탈 기준: 코스에서 30m 초과
- 복귀 기준: 코스에서 15m 이하
- 최소 유지 시간(예: 3초)과 히스테리시스를 적용해 깜빡임을 방지한다.

### 9.3 인증 정책
- MVP1에서는 인증을 단순화할 수 있다.
- 민감 데이터 보호를 염두에 두고, write 중심 기능은 이후 인증 도입을 전제로 설계한다.

### 9.4 위치 권한 정책
- 위치 권한 거부 시 기능 제한을 명확하게 안내한다.
- 백그라운드 위치는 MVP 필수 요구사항으로 두지 않는다.

---

## 10. 성공 지표

### 10.1 North Star Metric
- 주간 코스 따라가기 세션 수

### 10.2 보조 지표
- 라이딩 저장 → 코스 생성 전환율
- 코스 생성 대비 공유 링크 발급률
- 공유 링크 오픈율
- along-route POI 클릭/저장율
- Featured 코스 진입 후 따라가기 시작률

### 10.3 초기 검증 질문
- along-route POI가 실제 정차/우회 결정에 도움이 되는가?
- 코스 따라가기가 일반 지도 사용보다 낫다고 느끼는가?
- 사용자가 자발적으로 코스를 다시 열어보거나 공유하는가?

---

## 11. 리스크와 대응

### R1. along-route 가치가 약할 수 있음
- 대응: 일반 주변검색 대비 사용자 테스트/A-B 성격의 비교 실험 진행

### R2. off-route 오탐이 핵심 경험을 망칠 수 있음
- 대응: 정확도 표시, 히스테리시스, 최소 유지 시간 적용, 초기 도그푸딩으로 임계값 조정

### R3. 콜드스타트로 앱이 비어 보일 수 있음
- 대응: Featured 코스 최소 10개를 큐레이션해 선탑재

### R4. 외부 데이터 품질/동기화 실패
- 대응: 런타임 직접 호출 대신 동기화 후 DB 서빙, 실패 시 마지막 성공 데이터 유지

### R5. 위치 데이터 민감성
- 대응: 보관 기간, 로그 마스킹, 환경변수/시크릿 관리 정책 명시

## 14. 최종 정리

이 MVP의 본질은 단순한 지도 앱이 아니라, **경로 중심 편의 정보 + 코스 콘텐츠화 + 따라가기 경험**을 하나로 묶어 “가볍게 타는 자전거 사용자”에게 즉시 쓸모 있는 경험을 제공하는 것이다.

성공 판단은 기능 수가 아니라,
- 사용자가 코스를 실제로 따라가는지,
- along-route POI가 불안을 줄이는지,
- 기록이 코스 콘텐츠로 재사용되는지
로 결정한다.
