## 0. MVP3 착수 전 고정 체크리스트(결정)

아래 항목이 고정되면 구현에 착수한다(코드/DB는 문서와 동기화).

- 모임 참여 정책: 승인 없이 참여(선착순), 정원(capacity) 초과 시 참여 불가
- 최소 필드:
  - meetup: courseId, hostUserId, title, startAt, meetingPoint(lat/lon 선택), capacity(선택), status
  - participant: meetupId, userId, joinedAt
- 상태 모델:
  - meetup.status = open|closed|cancelled|finished
- 동시성(정원): Postgres row-level lock(SELECT ... FOR UPDATE) + 참가자 unique(meetup_id, user_id)
- API 초안(권장 경로):
  - POST `/api/v1/courses/{courseId}/meetups`
  - GET `/api/v1/courses/{courseId}/meetups?status=open`
  - GET `/api/v1/meetups/{meetupId}`
  - POST `/api/v1/meetups/{meetupId}/join`
  - POST `/api/v1/meetups/{meetupId}/leave`
- 채팅 정책:
  - 통신: WebSocket + STOMP
  - 보관: MVP3 초기에는 DB 저장하지 않음(임시 채팅)
  - 권한: 모임 참가자만 publish/subscribe 가능
  - 제한(초안): 메시지 길이 200자, 최근 N개만 유지(in-memory)
- 인증: MVP2의 서비스 JWT(Bearer) 기반으로 통일

검증/정책(초안):

- `startAt`은 현재 기준 과거 5분 이전 불가, 미래 180일 초과 불가
- `capacity`는 1~100 범위
- 모임장은 leave 불가(권한 오류)
- join/leave는 idempotent하게 처리(중복 join/이미 leave 상태는 성공 응답 가능)

## 1. 모임(Meetup)

- 코스 기준으로 모임을 생성/참여/탈퇴
- 정원 제한이 있으면 DB row lock으로 정합성 보장

스키마/API는 MVP3 착수 시 본 문서의 "고정 체크리스트"를 기준으로 확정한다.

### 1.1 API 계약(고정)

- POST `/api/v1/courses/{courseId}/meetups`
- GET `/api/v1/courses/{courseId}/meetups?status=open`
- GET `/api/v1/meetups/{meetupId}`
- POST `/api/v1/meetups/{meetupId}/join`
- POST `/api/v1/meetups/{meetupId}/leave`

create request 예시:

```json
{
  "title": "토요일 한강 9시",
  "startAt": "2026-03-07T09:00:00",
  "meetingPointLat": 37.518,
  "meetingPointLon": 126.938,
  "capacity": 8
}
```

list/detail response 예시:

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "meetupId": 1001,
    "courseId": 10,
    "title": "토요일 한강 9시",
    "status": "open",
    "startAt": "2026-03-07T09:00:00",
    "meetingPointLat": 37.518,
    "meetingPointLon": 126.938,
    "capacity": 8,
    "participantCount": 3,
    "joined": false,
    "host": false
  }
}
```

## 2. 임시 채팅(모임 그룹 채팅)

요구:

- 모임 단위 채팅방 1개
- 모임 종료 후 채팅은 보관하지 않거나(기본), 짧은 TTL로만 보관

통신:

- WebSocket + STOMP(Spring)
  - 장점: 방(토픽) 단위 pub-sub에 적합
  - 단점: 운영/스케일링 고려(브로커/세션)

저장 정책(권장):

- MVP3 초기: DB 저장 없이 in-memory + 제한된 메시지 수
- vNext: 저장이 필요해지면 `chat_messages` 테이블 + TTL/파티션 검토
