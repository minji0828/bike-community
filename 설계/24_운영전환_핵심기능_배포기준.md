# 운영 전환 핵심 기능 및 배포 기준

- 문서 ID: OPS-READY
- 버전: v0.1
- 작성일: 2026-03-07
- 상태: 초안

목적:

- 현재 로컬 중심 MVP를 실제 운영 가능한 수준으로 끌어올리기 위한 최소 기능과 배포 기준을 짧게 고정한다.
- 이후 백엔드/프론트 개발과 프론트 Vercel 배포의 기준 문서로 사용한다.

## 1. 우선순위

1. 인증/세션 안정화
2. 모임/채팅 운영 기능 보강
3. 코스/라이딩 핵심 사용자 흐름 완성
4. 여행 계획/안심 기능의 운영 기준 정리
5. 운영 배포/관측/장애 대응 정리

## 2. 운영 필수 기능

### 2.1 인증/세션

- 카카오 로그인 성공/실패 흐름을 웹에서 안정적으로 재현 가능해야 한다.
- 만료 토큰, 잘못된 토큰, 로그아웃 상태에서 사용자 메시지가 명확해야 한다.
- 프론트는 same-origin 인증 확인 경로와 백엔드 API 경로를 혼용하지 않는다.

### 2.2 모임/채팅

- 비참가자는 채팅방에 입장할 수 없어야 하며, 프론트는 연결 전에 이를 차단해야 한다.
- 참가자는 채팅방 진입 시 히스토리 1회 로드와 실시간 수신이 가능해야 한다.
- 운영 전환 전 최소 보강 항목:
  - 채팅 재연결 정책
  - 메시지 영속화 여부 결정
  - 신고/차단/모더레이션 정책
  - 모임 종료 후 채팅 보존 정책

### 2.3 코스/라이딩

- 코스 상세, 모임 생성/참가/탈퇴, 라이딩 저장, 코스화 흐름이 끊기지 않아야 한다.
- 실패 시점별 에러 메시지와 재시도 동선이 필요하다.

### 2.4 운영/관리

- 관리자용 최소 기능:
  - 신고 조회
  - 부적절 콘텐츠/채팅 대응
  - 장애 시 health/log 확인
- 최소 관측:
  - backend health
  - frontend health
  - 로그인 실패/채팅 실패 로그 추적


### 2.5 여행 계획/안심 기능

- Collection은 공개 범위/비노출/내부 코스 정합성이 보장되어야 한다.
- Highlight는 신고/숨김/만료 정책이 있어야 한다.
- Live Share는 만료형 링크/비공개 기본/종료 시 즉시 비활성화가 되어야 한다.
- Route Package는 stale data 고지가 있어야 한다.

## 3. 배포 기준

### 3.1 프론트 Vercel

- 프론트는 Vercel 배포를 기본 배포 경로로 사용한다.
- 필수 환경변수:
  - `NEXT_PUBLIC_API_BASE_URL`
  - `NEXT_PUBLIC_KAKAO_CLIENT_ID`
  - `NEXT_PUBLIC_KAKAO_REDIRECT_URI`
  - `NEXT_PUBLIC_KAKAO_MAP_JS_KEY`
- 배포 전 확인:
  - `npm run check`
  - 로그인 페이지/코스 상세/모임 채팅 라우트 정적 검증
  - 운영 도메인 기준 카카오 Redirect URI 일치

### 3.2 백엔드

- 백엔드는 현재 Docker + Spring Boot 기준을 유지한다.
- 운영 배포 전 필수 확인:
  - `/api/v1/health`
  - `/ws-stomp` 프록시 업그레이드
  - `APP_JWT_SECRET`, CORS, Kakao env
