## 0. 결정 사항

- 제품의 1차 기록 채널은 모바일 앱이다.
- MVP1은 foreground 위치 권한만 요구한다.
- 기본 샘플링은 `timeInterval=2000ms`, `distanceInterval=5m`다.
- 2m 미만 이동은 중복/노이즈로 보고 버린다.
- accuracy가 지나치게 나쁜 좌표는 버릴 수 있다. 초기 기준은 40m다.
- 업로드 전 path는 최대 250 포인트로 다운샘플링한다.
- 서버 저장 포맷은 PostGIS `LineString(SRID=4326)`다.

---

## 1. 모바일 UX 플로우

1. 사용자가 라이딩 기록을 시작한다.
2. 앱이 foreground 위치 권한을 확인한다.
3. 지도 위에 현재 위치와 누적 경로를 그린다.
4. 사용자가 기록을 종료한다.
5. 저장 API `POST /api/v1/ridings`를 호출한다.
6. 저장 성공 후 `POST /api/v1/courses/from-riding`로 코스 저장을 유도한다.

필수 표시:

- 경과 시간
- 누적 거리
- 기록 상태(recording / paused / saving)

---

## 2. 샘플링/필터링 정책

### 2.1 권한

- MVP1은 foreground 권한만 요구한다.
- 권한 거부 시 기록 기능은 제한하고, 코스 열람은 계속 허용한다.

### 2.2 좌표 수집

- 권장 기본값:
  - accuracy: balanced
  - timeInterval: 2000ms
  - distanceInterval: 5m

### 2.3 필터링

- 연속 포인트 간 거리가 2m 미만이면 skip
- accuracy가 40m 초과이면 skip 가능
- 비현실적 점프 속도는 skip 후보로 본다

### 2.4 다운샘플링

- 업로드 전 최대 250 포인트로 축소
- MVP1은 균등 샘플링으로 시작
- 추후 RDP 단순화는 후속 과제다

---

## 3. 서버 API 계약

### 3.1 라이딩 저장

- `POST /api/v1/ridings`
- 인증 필요
- `deviceUuid`, `path[]`는 필수
- 소유권은 JWT `sub`에서 결정한다

### 3.2 코스 저장

- `POST /api/v1/courses/from-riding`
- 인증 필요
- 본인 라이딩으로만 생성 가능

---

## 4. 웹 구현체와의 관계

- 현재 저장소 웹 앱은 브라우저 Geolocation API로 제한된 기록/가이드 경험만 제공한다.
- 웹 구현은 모바일 정책을 검증하기 위한 보조 수단이며, 제품 기준 GPS 정책의 SSOT는 이 문서다.
- 웹에서 백그라운드 기록이나 네이티브 센서 품질을 전제로 정책을 바꾸지 않는다.

---

## 5. 프라이버시/보안

- 라이딩 원본 좌표는 민감 정보로 취급한다.
- 일반 운영 로그에 위경도 원본을 남기지 않는다.
- 라이딩 원본은 직접 공유하지 않고, 코스 공유 링크를 기본 공유 단위로 사용한다.

---

## 6. 완료 기준

- 모바일에서 기록 시작/일시정지/종료가 동작한다.
- 저장 API 성공 시 `ridingId`를 받는다.
- 저장 직후 코스 생성으로 자연스럽게 이어진다.
- 생성된 코스를 모바일과 웹에서 공통 API로 조회할 수 있다.
