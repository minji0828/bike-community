## 0. 전제(중요)

- 로그인/회원가입은 당분간 넣지 않는다.
- 사용자는 `deviceUuid`로만 식별된다(신뢰도 낮음, 스푸핑 가능).
- 따라서 커뮤니티 기능은 "공개 피드"보다 "코스 중심" + "초대/코드 기반" 모델이 안전하다.

## 1. 커뮤니티 기능 후보(코스 중심)

### 1.1 코스에 사람을 모으는 3가지 축

1) 모집/같이 타기(모임)
   - 코스 기반으로 "이번 주 토요일 9시 같이 타요"를 만들고 참가자를 모집
2) 소셜 프루프(최근 탑승/사용 흔적)
   - "최근 7일 12명이 탔어요" 같은 신호(혼자라는 느낌 완화)
3) 코스 피드백(댓글/후기)
   - 위험/공사/좋은 포인트 같은 정보가 코스에 쌓임

MVP2 우선순위(권장):

- 1순위: 모집/같이 타기(모임) = 가장 강한 커뮤니티 체감
- 2순위: 댓글(코스 페이지가 '살아 있음'을 보여줌)
- 3순위: 소셜 프루프(탑승 집계) = 데이터 파이프라인이 필요

## 2. MVP2 제안: "코스 모임(Meetup)" 설계

### 2.1 핵심 UX

- 코스 상세에 "모임" 섹션
  - 진행 중 모집 리스트(날짜/시간/정원/장소)
  - "모임 만들기" CTA
- 모임은 기본적으로 "초대 코드(joinCode)"로 참여한다.
  - 로그인 없이도 링크/코드로 합류 가능

### 2.2 최소 데이터 모델(초안)

`course_meetups`

- id (PK)
- course_id (FK)
- host_device_uuid (not null)
- title (not null)
- start_at (timestamp)
- meeting_point geometry(Point,4326) (nullable)
- max_participants (int)
- participant_count (int, default 0)
- status (enum: open/closed/cancelled)
- join_code (unique)
- created_at, updated_at

`course_meetup_participants`

- meetup_id (FK)
- device_uuid (not null)
- nickname (nullable)
- role (enum: host/member)
- joined_at
- left_at (nullable)
- unique(meetup_id, device_uuid)

인덱스(권장)

- btree(course_id, start_at)
- btree(join_code)
- btree(meetup_id)

## 3. 동시성/DB Lock 설계(정원 모집이 핵심)

목표: "정원 초과" 없이 join을 처리하고, 재시도/중복 요청에도 안전해야 한다.

### 3.1 권장 패턴(1인 MVP2에 적절)

원칙:

- 분산락/Redis 없이 Postgres의 row-level lock으로 해결한다.
- join(정원 제한 + 중복 참가)이 포함되므로, "정확한 멱등성"이 중요하다.

권장 기본값:

- `SELECT ... FOR UPDATE`로 meetup row를 잠근 뒤,
  - 이미 참가자인지 확인
  - 정원 체크
  - 참가자 insert
  - 카운트 update
  를 트랜잭션으로 처리한다.

이유:

- 단순 원자 UPDATE만으로 join을 처리하면 "중복 join"에서 카운트가 틀어질 수 있어 주의가 필요하다.

상세 설계:

- `설계/20_MVP2_동시성_통신_락설계.md`

### 3.2 추가 안전장치(권장)

- 멱등성(Idempotency): 모바일 재시도/중복 탭을 고려
  - join 요청에 `Idempotency-Key` 헤더를 받아 24h 저장(선택)
- unique(meetup_id, device_uuid)로 중복 참가 방지
- leave 시 카운트 감소도 UPDATE로 안전하게 처리

## 4. 통신 방식 선택(모임/채팅/상태)

### 4.1 옵션

- 옵션 A: REST + 폴링(5~10초)
  - 장점: 구현 가장 단순, 운영 요소 최소
  - 단점: 실시간 체감 약함, 트래픽 증가

- 옵션 B: SSE(Server-Sent Events)
  - 장점: 서버->클라 단방향 실시간에 적합(상태 변화 알림)
  - 단점: 모바일 네트워크/백그라운드에서 연결 안정성 이슈 가능

- 옵션 C: WebSocket(STOMP)
  - 장점: 채팅/실시간 참여자 변화에 적합(양방향)
  - 단점: 인증/세션/스케일링 고려가 필요(향후 Redis 브로커 등)

권장 로드맵:

- MVP2-1: REST + 폴링으로 "모임"만 먼저 출시
- MVP2-2: 모임 채팅이 필요해지면 WebSocket 도입
- SSE는 "상태 알림" 용도로만 제한적으로 고려

## 5. Abuse/운영(로그인 없는 커뮤니티의 현실)

- 스팸/도배 방지(필수):
  - rate limit: 모임 생성/참여/댓글 작성에 deviceUuid 기준 제한
- 신고/차단(최소):
  - 코스/댓글/모임 신고 엔드포인트(운영이 숨김 처리)
- 닉네임 정책:
  - 기본은 "익명" + 랜덤 suffix(클라) 권장
