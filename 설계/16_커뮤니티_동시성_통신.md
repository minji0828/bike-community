## 0. 전제(중요)

- MVP2에서 카카오 소셜 로그인(OIDC)을 도입한다.
- 커뮤니티 write(댓글/신고)는 userId 기반으로 처리한다.
- 모임/채팅은 MVP3로 미룬다.

---

## 1. 커뮤니티 기능 후보(코스 중심)

### 1.1 코스에 사람을 모으는 3가지 축

1) 모집/같이 타기(모임)
   - 코스 기반으로 "이번 주 토요일 9시 같이 타요"를 만들고 참가자를 모집
2) 소셜 프루프(최근 탑승/사용 흔적)
   - "최근 7일 12명이 탔어요" 같은 신호(혼자라는 느낌 완화)
3) 코스 피드백(댓글/후기)
   - 위험/공사/좋은 포인트 같은 정보가 코스에 쌓임

MVP2 우선순위(권장):

- 1순위: 댓글(코스 페이지가 '살아 있음'을 보여줌)
- 2순위: 소셜 프루프(최근 탑승/활동) = 집계 파이프라인 필요
- 3순위: 모집/같이 타기(모임) = 동시성/채팅으로 MVP3에 적합

## 2. MVP2 제안: "코스 댓글(Comment)" 설계

### 2.1 핵심 UX

- 코스 상세에 "댓글" 섹션
  - 최신 댓글 리스트
  - 댓글 작성 CTA(로그인 유도)

### 2.2 최소 데이터 모델(초안)

`course_comments`

- id (PK)
- course_id (FK)
- author_user_id (not null)
- body (text)
- status (enum: visible/hidden/deleted)
- created_at, updated_at

인덱스:

- btree(course_id, created_at desc)
- btree(author_user_id, created_at desc)
- btree(status)

## 3. 동시성/DB Lock 설계(모임은 MVP3)

댓글은 강한 락이 필요 없다. 모임 정원/참여 동시성은 MVP3에서 다룬다.

MVP3 기본값(권장):

- Postgres row-level lock(`SELECT ... FOR UPDATE`) 기반 트랜잭션
- 참가자 중복 방지: `unique(meetup_id, user_id)`
- (선택) 모바일 재시도 대비 `Idempotency-Key`

상세 설계:

- `설계/22_MVP3_모임_채팅_설계.md`

## 4. 통신 방식 선택(모임/채팅/상태)

### 4.1 옵션

- 옵션 A: REST + 폴링(5~10초)
  - 장점: 구현 가장 단순, 운영 요소 최소
  - 단점: 실시간 체감 약함, 트래픽 증가

- 옵션 B: SSE(Server-Sent Events)
  - 장점: 서버->클라 단방향 실시간에 적합(상태 변화 알림)
  - 단점: 모바일 네트워크/백그라운드에서 연결 안정성 이슈 가능

- 옵션 C: WebSocket(STOMP)
  - 장점: 채팅/실시간 참여자 변화에 적합(양방향)
  - 단점: 인증/세션/스케일링 고려가 필요(향후 Redis 브로커 등)

권장 로드맵:

- MVP2: 댓글은 REST(재조회/새로고침)로 충분
- MVP3: 모임은 REST + 폴링(참여자/상태), 채팅은 WebSocket(STOMP)
- SSE는 "알림"(새 댓글/새 모임) 용도로만 제한적으로 고려

## 5. Abuse/운영(익명 표시 + 스팸 방지)

- 스팸/도배 방지(필수):
  - rate limit: 댓글 작성/신고는 userId 기준 제한
- 신고/차단(최소):
  - 코스/댓글/모임 신고 엔드포인트(운영이 숨김 처리)
- 닉네임 정책:
  - 댓글 작성자는 항상 "익명"으로 표시(식별 정보 노출 금지)

## 6. API 초안(커뮤니티)

댓글(MVP2)

- `POST /api/v1/courses/{courseId}/comments`
- `GET /api/v1/courses/{courseId}/comments`
- `DELETE /api/v1/comments/{commentId}`
- (선택) `POST /api/v1/comments/{commentId}/reports`

모임(MVP3)

- `POST /api/v1/courses/{courseId}/meetups`
- `GET /api/v1/courses/{courseId}/meetups?status=open`
- `POST /api/v1/meetups/join` (joinCode)
- `POST /api/v1/meetups/{meetupId}/leave`
- `GET /api/v1/meetups/{meetupId}`
