# Start Here (Big Guide): 설계 문서 운영/개발 플레이북

- 문서 ID: START-HERE
- 버전: v0.4
- 작성일: 2026-03-10
- 상태: 실행 가능한 설계 가이드(SSOT)

이 문서는 `설계/` 폴더 전체를 "개발이 바로 시작되는 수준"으로 묶어주는 단일 가이드다.
읽는 순서뿐 아니라, 문서를 어떻게 바꾸고(=결정), 어떻게 코드로 옮기고(=구현), 어떻게 검증하는지(=DoD)를 포함한다.

핵심 목표:

- 문서가 코드보다 먼저 결정된다(문서가 단일 진실원천, SSOT).
- MVP를 "바로 개발"할 수 있는 수준으로 범위/데이터/API/운영까지 끊김 없이 연결한다.
- L2(코스 따라가기)로 빨리 검증하고, vNext 추천/AI, 장기 TBT(레벨 4)로 확장한다.

## 1. Golden Rules (절대 규칙)

### 1.1 SSOT(단일 진실원천)

- MVP1 범위/플로우/용어/작업순서 변경: 먼저 `docs/01~05` 수정 -> 코드 수정
- API 변경: 먼저 `설계/11_API_명세.md` 수정 -> 코드 수정
  - (MVP2 확장) 인증/댓글은 `설계/21_인증_카카오_OIDC.md`, `설계/19_MVP2_커뮤니티_API_명세.md`도 함께 갱신
- DB 변경: 먼저 `설계/10_데이터모델_스키마.md` 수정 -> 마이그레이션/엔티티 수정
  - (MVP2 확장) 댓글/신고는 `설계/18_MVP2_커뮤니티_데이터모델_스키마.md`도 함께 갱신
- 정책 변경(공개범위/이탈 기준/태그/경고): 먼저 `설계/04_정책정의서.md` 수정 -> 코드/UI 수정
- 기술 선택 변경(프레임워크/지도 SDK/인증 흐름/실시간 통신/저장 전략): 먼저 ADR 작성 또는 수정 -> 관련 계약 문서 수정 -> 코드 수정

### 1.2 지오메트리 표준(좌표 순서)

실수하면 "바다에 찍히는" 수준의 치명적인 버그가 된다.

- DB/PostGIS/JTS(WKT 포함): 반드시 `(lon, lat)` = `(X, Y)` 순서를 유지한다.
- API JSON: 가독성을 위해 객체를 쓰되, 필드는 `{"lat": ..., "lon": ...}`로 둔다.
- API URL Query: `?lat=...&lon=...` 형태를 기본으로 한다(내부 생성은 항상 lon,lat).

관련 문서:

- 스키마/공간 질의: `설계/10_데이터모델_스키마.md`
- API 계약: `설계/11_API_명세.md`

### 1.3 MVP 범위 고정(스코프 관리)

- "MVP Must"가 아니면 지금 만들지 않는다.
- MVP Must는 반드시 USM의 "끝까지 연결"(featured -> 상세 -> 따라가기 / riding -> 코스 -> 공유)되는 형태여야 한다.

관련 문서:

- 스코프 고정: `설계/02_MVP_요구사항정의서.md`
- 우선순위/슬라이스: `설계/07_USM_사용자스토리맵.md`

### 1.4 작업 종료 Git 규칙

- 작업이 끝나고 검증(테스트/빌드)이 통과하면, 해당 작업 단위로 커밋한다.
- 원격 브랜치에 즉시 push하여 작업 상태를 동기화한다.
- 커밋 단위는 "한 기능/한 수정 목적"을 원칙으로 한다.

### 1.5 작업 결과 보고(Work Report) 규칙

- 모든 작업 단위(테스트 통과 + 커밋/푸시)마다 `work-reports/`에 작업 일지(보고서)를 1개 남긴다.
- 파일명 규칙: `YYYY-MM-DD_HHMM_<짧은제목>.md` (KST 기준, 제목은 되도록 한글)
- 보고서에는 최소 다음을 포함한다:
  - 목표/배경(왜)
  - 완료된 항목(무엇)
  - 변경점(주요 파일/커밋)
  - 검증(실행한 테스트/빌드)
  - 다음 작업(Next)
  - 마일스톤/진척도: `설계/14_개발계획_마일스톤_체크리스트.md` 기준으로 현재 상태 + %
- 시크릿/토큰/개인정보(정확한 위치 포함)는 보고서에 절대 기록하지 않는다.

## 5. "바로 개발 가능" 판정 기준

다음이 Yes면 개발 시작해도 된다:

- USM의 MVP 컷이 명확하다: `설계/07_USM_사용자스토리맵.md`

- 제품 기준 모바일과 현재 repo 웹 구현체의 역할 분리가 명시되어 있다.
- 신규 스키마(코스/태그/경고)가 enum/상태값까지 고정되어 있다: `설계/10_데이터모델_스키마.md`
- 신규 API 형태가 auth/validation/error까지 고정되어 있다: `설계/11_API_명세.md`
- 런타임 플로우에서 "동기 vs 비동기"가 구분되어 있다: `설계/12_아키텍처_런타임플로우.md`
- 개발 순서와 DoD가 있다: `설계/14_개발계획_마일스톤_체크리스트.md`

### 5.1 현재 판정(2026-03-10)

- 제품 기획/범위: 대체로 준비됨
- 백엔드 핵심 MVP: 준비도 높음. 다만 API/스키마 문서의 초안 표현을 더 줄여야 한다.
- 프론트엔드: 모바일 제품 방향은 명확해졌고, 현재 repo 웹 구현체는 별도 가이드로 분리됐다.
- AI 추천/에이전트: 전략 문서는 있으나 구현 계약(I/O, 평가셋, 실패 정책)은 아직 없다.

   - 근거: 코스가 자산이며 featured/따라가기/추천이 모두 코스에서 시작

2) 메타데이터 파생치(distance/bbox/loop/toilet_count) 파이프라인을 만든다
   - 근거: 추천/표시/품질이 메타데이터에 달려있음

3) featured 코스를 최소 10개 시드로 준비한다(운영 API 또는 수동 시드)
   - 근거: 콜드스타트 해결
   - 권장: GPX 원본을 함께 보관해 재다운로드/재공유 가능하게 한다

4) 프론트엔드는 featured -> 상세 -> 따라가기(L2 또는 현재 저장소 기준 동등 UX)만 먼저 끝까지 연결한다
   - 근거: 가장 빨리 "쓸모"를 검증할 수 있음

상세 마일스톤/DoD:

- `설계/14_개발계획_마일스톤_체크리스트.md`

## 8. 피처드 코스 시드(콜드스타트) 운영 방식(초안)

목표: MVP에서 앱이 비어있지 않게 "기본 코스"를 제공한다.

권장 방식(구현 난이도 순):

1) DB 직접 시드(SQL/스크립트)
2) admin API로 코스 생성/수정 후 featuredRank 설정
3) (추후) GPX/GeoJSON 업로드 지원

시드용 최소 JSON 형식(권장, 1개 코스):

```json
{
  "title": "한강 산책(초보)",
  "visibility": "public",
  "sourceType": "curated",
  "region": "seoul",
  "tags": ["river", "beginner"],
  "warnings": [
    { "type": "busyCrossing", "severity": 2, "note": "교차로 주의" }
  ],
  "path": [
    { "lat": 37.52, "lon": 126.94 },
    { "lat": 37.53, "lon": 126.95 }
  ]
}
```

주의:

- 내부 저장은 반드시 (lon,lat)로 변환한다.
- 공유/공개 코스는 프라이버시 클리핑 정책을 적용한다(정책 문서 참조).
