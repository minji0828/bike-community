# Start Here (Big Guide): 설계 문서 운영/개발 플레이북

- 문서 ID: START-HERE
- 버전: v0.2
- 작성일: 2026-03-04
- 상태: 실행 가능한 설계 가이드(SSOT)

이 문서는 `설계/` 폴더 전체를 "개발이 바로 시작되는 수준"으로 묶어주는 단일 가이드다.
읽는 순서뿐 아니라, 문서를 어떻게 바꾸고(=결정), 어떻게 코드로 옮기고(=구현), 어떻게 검증하는지(=DoD)를 포함한다.

핵심 목표:

- 문서가 코드보다 먼저 결정된다(문서가 단일 진실원천, SSOT).
- MVP를 "바로 개발"할 수 있는 수준으로 범위/데이터/API/운영까지 끊김 없이 연결한다.
- L2(코스 따라가기)로 빨리 검증하고, vNext 추천/AI, 장기 TBT(레벨 4)로 확장한다.

## 1. Golden Rules (절대 규칙)

### 1.1 SSOT(단일 진실원천)

- API 변경: 먼저 `설계/11_API_명세.md` 수정 -> 코드 수정
- DB 변경: 먼저 `설계/10_데이터모델_스키마.md` 수정 -> 마이그레이션/엔티티 수정
- 정책 변경(공개범위/이탈 기준/태그/경고): 먼저 `설계/04_정책정의서.md` 수정 -> 코드/UI 수정

### 1.2 지오메트리 표준(좌표 순서)

실수하면 "바다에 찍히는" 수준의 치명적인 버그가 된다.

- DB/PostGIS/JTS(WKT 포함): 반드시 `(lon, lat)` = `(X, Y)` 순서를 유지한다.
- API JSON: 가독성을 위해 객체를 쓰되, 필드는 `{"lat": ..., "lon": ...}`로 둔다.
- API URL Query: `?lat=...&lon=...` 형태를 기본으로 한다(내부 생성은 항상 lon,lat).

관련 문서:

- 스키마/공간 질의: `설계/10_데이터모델_스키마.md`
- API 계약: `설계/11_API_명세.md`

### 1.3 MVP 범위 고정(스코프 관리)

- "MVP Must"가 아니면 지금 만들지 않는다.
- MVP Must는 반드시 USM의 "끝까지 연결"(featured -> 상세 -> 따라가기 / riding -> 코스 -> 공유)되는 형태여야 한다.

관련 문서:

- 스코프 고정: `설계/02_MVP_요구사항정의서.md`
- 우선순위/슬라이스: `설계/07_USM_사용자스토리맵.md`

   - 근거: 가장 빨리 "쓸모"를 검증할 수 있음

## 8. 피처드 코스 시드(콜드스타트) 운영 방식(초안)

목표: MVP에서 앱이 비어있지 않게 "기본 코스"를 제공한다.

권장 방식(구현 난이도 순):

1) DB 직접 시드(SQL/스크립트)
2) admin API로 코스 생성/수정 후 featuredRank 설정
3) (추후) GPX/GeoJSON 업로드 지원

시드용 최소 JSON 형식(권장, 1개 코스):

```json
{
  "title": "한강 산책(초보)",
  "visibility": "public",
  "sourceType": "curated",
  "region": "seoul",
  "tags": ["river", "beginner"],
  "warnings": [
    { "type": "busyCrossing", "severity": 2, "note": "교차로 주의" }
  ],
  "path": [
    { "lat": 37.52, "lon": 126.94 },
    { "lat": 37.53, "lon": 126.95 }
  ]
}
```

주의:

- 내부 저장은 반드시 (lon,lat)로 변환한다.
- 공유/공개 코스는 프라이버시 클리핑 정책을 적용한다(정책 문서 참조).
