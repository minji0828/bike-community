# ADR 운영 가이드

- 버전: v1.0
- 작성일: 2026-03-10
- 상태: 활성

ADR(Architecture Decision Record)은 기술 선택의 판단 근거와 결과를 남기는 기록이다.  
기능 요구사항 문서가 "무엇을 만들지"를 고정한다면, ADR은 "왜 이 기술 선택을 했는지"를 고정한다.

## 1. 폴더 구조

- `설계/adr/backend/`: 백엔드 런타임, 저장소, 인증, 통신, 운영 관련 결정
- `설계/adr/frontend/`: 프론트엔드 플랫폼, 라우팅, 상태관리, 인증, 지도, 실시간 통신 관련 결정

## 2. 상태 값

- `Proposed`: 제안만 되었고 아직 채택되지 않음
- `Accepted`: 현재 기준 채택됨
- `Superseded`: 다른 ADR로 대체됨
- `Deprecated`: 신규 작업 기준으로 더 이상 권장하지 않음
- `Rejected`: 검토했지만 채택하지 않음

## 3. 언제 ADR을 추가하나

다음에 해당하면 ADR을 먼저 작성하거나 갱신한다.

- 프레임워크, SDK, 외부 인프라를 새로 도입한다
- 인증 방식, API transport, 상태관리 방식이 바뀐다
- 데이터 저장 전략, 캐시 전략, 실시간 통신 전략이 바뀐다
- 앞으로 구현자가 되묻게 될 만한 기술 선택이 생긴다

## 4. 언제 ADR이 필요 없는가

- 단순 구현 상세
- 코드 스타일 수준의 선택
- 이미 Accepted ADR 안에서 자연스럽게 파생되는 작은 결정

## 5. 작성 규칙

- 번호는 `0001`부터 시작하고 백엔드/프론트엔드는 별도 시퀀스로 관리한다.
- 파일명은 `0001-short-kebab-case.md` 형식을 사용한다.
- 한 ADR에는 하나의 결정만 담는다.
- `Context`, `Options Considered`, `Decision`, `Rationale`, `Consequences`, `Revisit When`은 반드시 쓴다.
- 채택안만 적지 말고 최소 2개 이상 대안과 포기 이유를 함께 남긴다.
- `Rationale`에는 장점만이 아니라 트레이드오프를 명시한다.
- 결정이 바뀌면 기존 ADR을 수정해 덮지 말고 새 ADR을 만들고 이전 ADR 상태를 `Superseded`로 바꾼다.

## 6. 구현 문서와의 관계

- 제품/정책/API/스키마 계약은 여전히 `설계/04`, `10`, `11`, `21`, `23`, `29`가 우선이다.
- ADR은 그 계약을 뒷받침하는 기술 판단 기록이다.
- 구현 계약과 ADR이 충돌하면 먼저 ADR을 갱신한 뒤 계약 문서를 맞춘다.
