# BE-ADR-0002: 공간 연산 경계와 좌표 정책

- Status: Accepted
- Date: 2026-03-10

## Context

- 코스, 라이딩, POI는 모두 좌표와 geometry를 다루며 작은 좌표 실수도 치명적인 버그를 만든다.
- 클라이언트는 지도 렌더링을 위해 좌표를 쓰지만, 거리 계산/교차/주변 검색은 서버 책임이 더 적절하다.
- 현재 문서와 코드 모두 좌표 표현이 섞일 위험이 있다.

## Decision

- 서버는 geometry 저장, 공간 질의, 메타데이터 계산의 단일 책임을 가진다.
- DB/PostGIS/JTS/WKT 내부 표준 좌표 순서는 항상 `(lon, lat)`로 고정한다.
- API DTO는 가독성을 위해 `{ "lat": ..., "lon": ... }`를 사용한다.
- 프론트 내부 뷰 모델에서 `lng`를 쓰더라도 API 경계에서만 `lon <-> lng` 변환을 수행한다.
- `nearby`, `along-route`, bbox, 거리 계산, privacy clipping, amenities summary 같은 공간 연산은 서버가 담당한다.

## Rationale

- 좌표 순서를 한 곳에서만 바꾸게 해야 실수가 줄어든다.
- PostGIS/JTS는 공간 연산 정확도와 표현력이 충분하다.
- 클라이언트가 공간 검색까지 책임지면 구현 중복과 배터리/성능 문제가 커진다.
- API에서는 `lat`, `lon`이 사람이 읽고 디버깅하기 쉽다.

## Consequences

- 장점:
  - 공간 버그를 줄이고 테스트 경계를 명확히 할 수 있다.
  - 메타데이터 재계산과 검색 로직이 서버에 모여 일관성이 생긴다.
  - 클라이언트는 주로 렌더링과 현재 위치 추적에 집중할 수 있다.
- 단점:
  - 프론트/백엔드 모두 좌표 변환 규칙을 의식해야 한다.
  - 공간 연산 부담이 서버와 DB에 집중된다.
- 구현 시 주의점:
  - DTO, Entity, Mapper 테스트에서 좌표 순서 검증을 반드시 넣는다.
  - 새 API를 만들 때 query param도 `lat`, `lon` 표기를 유지한다.

## Revisit When

- 오프라인 우선 모바일 앱에서 대규모 로컬 공간 검색이 필수가 될 때
- 별도 지도/공간 검색 서비스로 분리할 필요가 생길 때
