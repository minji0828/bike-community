# BE-ADR-0001: Spring Boot 기반 모듈러 모놀리스 + Postgres/PostGIS

- Status: Accepted
- Date: 2026-03-10

## Context

- 현재 팀 규모와 저장소 구조상 빠른 기능 반복과 일관된 배포가 더 중요하다.
- 코스, 라이딩, POI, 인증, 모임, 채팅은 서로 강하게 연결되어 있고 같은 트랜잭션 경계가 필요한 경우가 많다.
- 이 프로젝트는 공간 질의와 경로 데이터를 많이 다루므로 일반 RDB만으로는 구현 비용이 커진다.

## Decision

- 백엔드는 단일 배포 단위의 Spring Boot 애플리케이션으로 유지한다.
- 패키지 구조는 `com.bikeoasis.domain.*` 중심의 모듈러 모놀리스로 유지한다.
- 런타임/언어 표준은 Java 17 + Spring Boot 3.5.x로 고정한다.
- 영속 계층은 PostgreSQL + PostGIS + Spring Data JPA + Hibernate Spatial + JTS 조합을 사용한다.
- 공개 API는 `/api/v1` 하위의 HTTP JSON 계약으로 유지한다.

## Rationale

- 지금 단계에서 마이크로서비스보다 모듈러 모놀리스가 변경 비용과 배포 복잡도를 낮춘다.
- PostGIS는 `nearby`, `along-route`, 코스 geometry 저장과 검색 같은 핵심 요구사항을 자연스럽게 지원한다.
- Spring Boot + JPA는 현재 코드베이스와 테스트 구조에 이미 맞춰져 있다.
- 도메인별 패키지 분리는 단일 앱 안에서도 책임 경계를 유지하게 해준다.

## Consequences

- 장점:
  - 기능 추가 시 트랜잭션, 보안, DTO 규칙을 한 애플리케이션 안에서 빠르게 일관되게 유지할 수 있다.
  - 운영 배포, 로컬 실행, 테스트 흐름이 단순하다.
  - 공간 질의를 DB 레벨에서 처리할 수 있어 구현 난이도가 낮아진다.
- 단점:
  - 앱 전체를 함께 배포해야 하므로 서비스 단위 독립 배포는 어렵다.
  - 내부 경계를 코드 규칙으로 지켜야 하며, 경계가 느슨해지면 모놀리스가 비대해질 수 있다.
- 구현 시 주의점:
  - 도메인 간 호출은 명확한 서비스/DTO 경계로 유지한다.
  - 새 기능을 추가할 때도 먼저 별도 서비스로 분리하고 `global`로 기능을 밀어넣지 않는다.

## Revisit When

- 팀이 다수의 독립 배포 단위가 필요한 수준으로 커질 때
- 특정 도메인이 독립된 확장/배포/보안 요구를 강하게 갖게 될 때
- 단일 DB와 단일 앱이 명확한 병목이 될 때
