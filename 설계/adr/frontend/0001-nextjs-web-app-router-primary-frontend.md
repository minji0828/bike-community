# FE-ADR-0001: 현재 저장소의 프론트엔드 구현체는 Next.js App Router 웹 앱

- Status: Superseded by FE-ADR-0006
- Date: 2026-03-10

## Context

- 설계 문서 일부는 RN/Expo를 전제로 작성되어 있지만, 2026-03-10 시점 저장소의 실제 프론트엔드는 `frontend/` 하위 Next.js 앱이었다.
- 인증, 지도 SDK, 채팅, 딥링크 정책은 플랫폼 전제가 다르면 구현 판단도 달라진다.
- 구현 시 되묻는 일을 줄이려면 "지금 당장 기준이 되는 프론트엔드"를 명확히 해야 한다.

## Decision

- 현재 저장소에서 실제로 빌드/배포되는 프론트엔드 구현체는 Next.js App Router 웹 앱으로 기록한다.
- 이 결정은 "제품의 1차 클라이언트가 무엇인가"를 뜻하지 않고, 당시 저장소 현실을 설명하는 구현 기록으로만 남긴다.
- 제품 관점의 1차 클라이언트 판단은 FE-ADR-0006에서 다시 고정한다.

## Rationale

- 코드와 문서의 플랫폼 전제가 다르면 구현 품질보다 추측 비용이 커진다.
- Next.js App Router는 현재 라우팅, app route, 인증 프록시, 클라이언트 컴포넌트 사용 패턴과 맞는다.
- 다만 이 판단만으로 제품 플랫폼을 웹으로 확정하면, 모바일 중심 PRD와 충돌한다.

## Consequences

- 장점:
  - 현재 저장소에서 구현을 시작할 때 플랫폼 질문이 줄어든다.
  - 웹 라우팅, 브라우저 위치 API, Kakao Maps JS SDK를 자연스럽게 사용한다.
  - 배포 타깃과 운영 기준을 Vercel/웹 기준으로 정리할 수 있다.
- 단점:
  - 제품 방향을 웹 우선으로 오해하게 만들 수 있다.
  - 모바일 재착수 시 별도 설계 트랙이 필요하다.

## Revisit When

- 제품의 1차 클라이언트를 명시적으로 고정할 때
- `frontend/mobile` 같은 실제 모바일 클라이언트가 다시 생길 때
- 웹과 모바일을 동등 1차 타깃으로 운영하게 될 때
