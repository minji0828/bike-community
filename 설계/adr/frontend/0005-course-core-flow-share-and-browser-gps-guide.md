# FE-ADR-0005: 코스 핵심경로는 공유 링크와 브라우저 GPS 가이드를 우선한다

- Status: Accepted
- Date: 2026-03-10

## Context

- 현재 MVP1의 핵심 가설은 사용자가 코스를 보고, 따라가고, 다른 사람에게 공유하는 흐름이 실제로 동작하는지 검증하는 것이다.
- 기존 프론트는 코스 상세에 확장 기능이 많고, 가이드 화면은 타이머 시뮬레이션에 가까워 핵심 경로 검증력이 약했다.
- 현재 프론트 기준 구현체는 Next.js 웹 앱이므로 네이티브 전용 GPS SDK나 복잡한 실시간 인프라를 전제로 하면 구현과 운영 부담이 커진다.

## Options Considered

- 옵션 A: 상세 화면에서 컬렉션, 하이라이트, 모임 같은 확장 기능을 계속 전면에 두고 가이드는 mock 상태로 유지한다.
  - 장점: 화면이 풍성해 보이고 기존 구현을 많이 버리지 않아도 된다.
  - 단점: MVP1 핵심경로 검증이 흐려지고 실제 따라가기 품질을 판단할 수 없다.
  - 포기 이유: 제품 검증 우선순위와 맞지 않는다.
- 옵션 B: 코스 상세의 최상단 행동을 `따라가기`, `공유`로 고정하고, 가이드는 브라우저 `watchPosition`과 경로 투영 계산으로 구현한다.
  - 장점: 웹 환경에서 바로 동작 가능한 최소 구현으로 핵심경로를 검증할 수 있다.
  - 단점: 네이티브 수준의 센서 품질이나 백그라운드 추적은 기대하기 어렵다.
  - 채택 이유: 현재 저장소와 MVP1 목표에 가장 적합한 구현 전략이다.
- 옵션 C: React Native/네이티브 GPS SDK를 다시 기준 구현체로 되돌린다.
  - 장점: 장기적으로 위치 추적 품질은 좋아질 수 있다.
  - 단점: 현재 코드베이스, 배포 환경, 문서 체계를 다시 흔들게 된다.
  - 포기 이유: 지금 단계에서 비용이 너무 크고 구현 일관성이 깨진다.

## Decision

- 코스 상세 화면의 1차 CTA는 `따라가기`와 `공유 링크 만들기`로 고정한다.
- 공개 공유는 `/share/[shareId]` 읽기 전용 경로와 public course API를 사용한다.
- 가이드 화면은 브라우저 geolocation `watchPosition`을 사용하고, 클라이언트에서 경로 투영으로 진행률과 이탈 여부를 계산한다.
- 초기 이탈 판정은 경로로부터 30m 초과가 3회 연속 감지될 때로 둔다.
- 컬렉션, 하이라이트, 코스모임 같은 확장 기능은 유지하되 P1 실험 영역으로 뒤로 보낸다.

## Rationale

- 사용자와 구현자 모두가 가장 먼저 보게 되는 화면에서 핵심 행동이 드러나야 문서와 제품이 일치한다.
- 공개 공유 링크와 브라우저 GPS는 별도 앱 설치나 네이티브 브리지 없이도 웹에서 검증 가능한 최소 완성선이다.
- 네이티브 수준의 추적 정밀도는 포기하는 트레이드오프가 있지만, 현재 단계에서는 구현 가능성과 검증 속도가 더 중요하다.

## Consequences

- 장점:
  - 코스 상세 -> 가이드 -> 공유 흐름이 실제 동작하는 MVP1 중심 UX가 된다.
  - 프론트 구현 복잡도를 과도하게 높이지 않고도 실사용 검증이 가능하다.
  - 공개 링크로 비로그인 사용자 유입 테스트가 쉬워진다.
- 단점:
  - 브라우저 geolocation 품질은 기기와 브라우저에 따라 편차가 있다.
  - 백그라운드 추적, 음성 내비게이션, 고급 오프라인 기능은 아직 지원하지 않는다.
- 구현 시 주의점:
  - 코스 상세는 silent sample fallback으로 실제 API 누락을 숨기지 않는다.
  - 가이드의 이탈 기준, 진행률 계산식, 위치 권한 실패 UX는 문서와 코드에서 함께 관리한다.
  - 확장 기능을 전면에 다시 올릴 때는 핵심 CTA 가시성을 해치지 않도록 새 ADR 또는 문서 갱신을 먼저 한다.

## Revisit When

- 네이티브 앱이 다시 1차 구현체가 될 때
- 백그라운드 GPS, 음성 안내, 오프라인 맵이 필수 요구사항이 될 때
- 공유 링크의 편집 권한, 만료, 권한 범위 정책이 복잡해질 때
