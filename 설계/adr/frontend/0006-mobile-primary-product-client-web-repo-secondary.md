# FE-ADR-0006: 제품의 1차 클라이언트는 모바일이며 현재 저장소 웹 앱은 보조 구현체다

- Status: Accepted
- Date: 2026-03-11

## Context

- PRD와 초기 설계 문서의 핵심 사용 시나리오는 GPS 기반 라이딩 기록, 모바일 위치 권한, 공유 링크 진입, 현장 따라가기 경험을 중심으로 짜여 있다.
- 반면 2026-03-11 현재 저장소에는 `frontend/mobile` 구현이 없고, `frontend/` 하위 Next.js 웹 앱만 존재한다.
- 이 상태를 문서에 명시하지 않으면 "제품은 모바일 기반인데 설계는 웹으로 잠기고", "현재 저장소는 웹인데 구현자는 모바일 코드가 있다고 오해하는" 이중 혼선이 생긴다.

## Decision

- 제품 관점의 1차 클라이언트는 모바일 앱으로 고정한다.
- 모바일 앱의 기준 구현 스택은 React Native + Expo 계열로 유지하고, 실제 소스는 추후 `frontend/mobile` 트랙으로 관리한다.
- 현재 저장소의 `frontend/` Next.js 앱은 보조 구현체이자 운영/검증/공유 진입용 웹 클라이언트로 취급한다.
- 문서 레이어는 아래처럼 분리한다.
  - 모바일 제품 설계/클라이언트 계약: `설계/35_모바일_클라이언트_개발_가이드.md`
  - 현재 저장소 웹 구현 규칙: `설계/29_프론트엔드_개발_가이드.md`

## Options Considered

### A. 웹을 1차 클라이언트로 계속 고정

- 장점:
  - 현재 저장소와 가장 잘 맞는다.
  - 웹 배포/운영 문서와 일관된다.
- 단점:
  - PRD의 모바일 가치 제안과 충돌한다.
  - GPS 권한, 현장 따라가기, 앱 설치형 사용성 검증 관점이 약해진다.

### B. 모바일을 1차 클라이언트로 두고 현재 웹은 보조 구현체로 분리

- 장점:
  - 제품 방향과 저장소 현실을 동시에 설명할 수 있다.
  - 모바일 설계와 웹 구현 가이드를 충돌 없이 병행 관리할 수 있다.
- 단점:
  - 문서가 한 장으로 끝나지 않고 플랫폼별 가이드가 필요하다.
  - 실제 모바일 소스가 없기 때문에 구현 착수 전 초기 셋업 비용이 남는다.

## Rationale

- 서비스 핵심 가치인 "기록하면서 따라가기"는 모바일 디바이스가 자연스러운 1차 사용 환경이다.
- 현재 저장소에는 웹 구현체가 있으므로, 이를 무시하면 검증 가능한 자산을 버리게 된다.
- 따라서 "제품 1차 플랫폼"과 "현재 저장소 구현 현실"을 분리해 기록하는 것이 가장 정직하고 유지보수 가능하다.

## Consequences

- 장점:
  - PRD, SDD, 현재 코드가 서로 다른 층위의 진실을 다룬다는 점을 명확히 할 수 있다.
  - 모바일 착수 시 재질문 없이 기본 경계를 사용할 수 있다.
  - 웹 앱은 공유 링크, 운영 검증, 공개 코스 열람에 계속 활용할 수 있다.
- 단점:
  - 문서 독자가 "어느 문서를 먼저 봐야 하는가"를 분명히 안내받아야 한다.
  - 모바일 빌드/배포 자동화는 별도 과제로 남는다.

## Revisit When

- 실제 `frontend/mobile` 코드베이스가 생성될 때
- 모바일과 웹의 기능 범위가 달라질 때
- 모바일이 아니라 웹을 제품의 1차 채널로 전환하기로 결정할 때
