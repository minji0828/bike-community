## POL-001 인증/권한

- MVP1 현재 기준:
  - 읽기 API는 기본 공개를 유지한다.
  - 사용자 생성 write와 개인 데이터 접근은 인증이 필요하다.
- MVP1 인증 필수 대상(현재 기준):
  - `POST /api/v1/ridings`
  - `POST /api/v1/courses`
  - `POST /api/v1/courses/from-riding`
  - `POST /api/v1/courses/gpx`
  - `POST /api/v1/courses/{courseId}/share`
- 소유권 정책:
  - 라이딩/코스 생성과 공유 발급의 소유권은 요청 바디가 아니라 JWT `sub`에서 확정한다.
  - 타인 소유 riding/course에 대한 write는 `403`으로 차단한다.
- 추후:
  - 사용자 본인 데이터 접근만 허용
  - 관리자(backoffice) 권한 분리

## POL-002 위치 데이터 취급

- 위치는 개인정보(민감)로 간주
- 로그 정책:
  - 운영에서는 위경도 원본을 INFO로 남기지 않는다
  - DEBUG도 필요 최소화(마스킹/샘플링)

- 기본 보관 기간: 30일 (변경 가능)
- 삭제 작업:

  - 삭제 결과(삭제 건수) 로그

- 모드: `full` 또는 `incremental`

  - 기동 시 동기화 실패가 서비스 전체를 막지 않도록 옵션화
  - (추후) 관리자에게 알림/대시보드 표시

## POL-007 코스 공개 범위

- `public`: 누구나 조회 가능
- `unlisted`: 링크를 아는 사람만 조회 가능
- `private`: 소유자만 조회 가능

MVP 권장 기본값:

- 코스 생성 시 기본은 `private`
- 공유 링크를 만들면 `unlisted` 또는 `public`로 전환(정책 선택)

## POL-008 길안내 레벨 정책

- MVP는 레벨 2(코스 따라가기 가이드)만 제공한다.
- 턴바이턴(레벨 4)은 별도 단계로 계획하고, 구현 전 아래를 확정한다:
  - 라우팅/내비 제공 방식(외부 앱 호출 vs 내장 SDK vs 자체)
  - 비용/쿼터/약관(지도/길찾기/음성)
  - 개인정보/보안(정확한 경로 공유)

## POL-009 위치 권한/백그라운드

- 홈과 코스 상세는 위치 권한이 없어도 읽기 위주로 계속 사용할 수 있다.
- 라이딩 기록과 실시간 가이드는 위치 권한이 필요하며, 거부 시 제한 모드와 안내 문구를 명확히 고지한다.
- 백그라운드 위치는 MVP에서는 필수로 요구하지 않는다(필요 시 별도 동의/고지).

## POL-010 코스 이탈(off-route) 판정

- 이탈 기준은 "코스 선과의 거리"로 판정한다.
- MVP1 현재 기준:
  - 이탈: 코스에서 30m 초과가 3회 연속 관측되면 `off_route`
  - 복귀: 다시 30m 이하로 관측되면 `playing`으로 복귀
- 메모:
  - 더 강한 히스테리시스(예: 복귀 15m)는 운영 데이터 확인 후 vNext에서 재검토한다.

## POL-011 코스 메타데이터 v1(필수/자동/입력)

원칙:

- 코스 추천/검색을 위해 핵심 메타데이터는 구조화한다.
- 사용자 입력 부담을 줄이기 위해 파생치(거리/예상시간/POI 요약)는 서버가 자동 산출한다.
- 코스 geometry는 SRID 4326 + 좌표 순서 (lon,lat)을 표준으로 한다.

필수(코스 생성 시):

- `title`
- `path`(LineString)
- `visibility`
- `sourceType` (curated/ugc/import)

자동 산출(서버):

- `distanceKm`, `bbox`, `loop`, `estimatedDurationMin`
- `amenitiesSummary` (toiletCount/cafeCount 등)

선택 입력(사용자/큐레이터):

- `tags[]`, `notes`, `warnings[]`, `highlights[]`

## POL-012 태그 사전 정책

목표: 태그가 많아질수록 추천 품질이 좋아지지만, 난잡해지면 입력/검색 품질이 무너진다.

- 태그는 "자유 입력"이 아니라 사전 기반 멀티 선택을 기본으로 한다.
- 사전은 백오피스에서 관리한다(표시명, 카테고리, 비활성화).
- MVP 최소 태그 세트(권장, 수정 가능):
  - 분위기: `한적`, `도심`, `강변`, `공원`
  - 난이도/목적: `초보`, `운동`, `산책`, `업힐`
  - 동행: `가족`, `혼자`

## POL-013 경고/하이라이트(특이사항) 정책

정의:

- 경고(warnings): 위험/불편/주의가 필요한 정보
- 하이라이트(highlights): 뷰포인트/정차 추천/포인트

구조(권장):

- 경고 type 예시: `busyCrossing | steepDescent | gravel | narrowLane | construction | noLight`
- 심각도(severity): 1~3
- 위치(at)와 반경(radiusM)은 선택(코스 전체 주의는 위치 없이 작성 가능)
- 공사/임시 이슈는 validUntil로 만료를 지정할 수 있다

고지:

- 경고/하이라이트는 "참고 정보"이며 완전성을 보장하지 않는다.
- 실제 주행 안전 책임은 사용자에게 있다(앱 내 고지 문구 필요).

## POL-014 큐레이션/피처드 코스 정책

- 기본 제공 코스는 sourceType=`curated`, verifiedStatus=`curated`로 관리한다.
- 피처드 노출(featuredRank)은 운영/큐레이터가 결정한다.
- 안전/공사 등 이슈가 있으면 즉시 비노출 처리 가능해야 한다.

## POL-015 코스 공유 프라이버시(초안)

문제: 공유 코스는 생활반경(집/직장) 노출 위험이 있다.

MVP1 현재 기준:

- 기본 visibility는 `private`
- 공유 링크 발급은 `public` 또는 `unlisted` 코스에만 허용한다.
- `private` 코스는 공유 링크로 직접 노출하지 않는다.
- 코스가 삭제되거나 `private`로 전환되면 기존 공유 링크는 무효화한다.
- 공유 코스는 읽기 전용 진입을 기본으로 하며, 쓰기 기능은 로그인/소유권 확인 뒤에만 노출한다.

출발/도착 프라이버시 보강(vNext 후보):

- path clipping, 샘플링, 시작점 블러는 운영 전 보강 후보로 유지한다.
- 현재 문서 기준 MVP1 필수 완료 조건에는 path clipping을 넣지 않는다.

vNext 후보:

- 홈 반경 마스킹: 시작/끝 구간을 X m 잘라내거나(clip) 출발지 주변을 흐림 처리

## POL-016 공유 링크 무효화(삭제/비노출)

- 코스가 삭제되거나 비노출(운영) 처리되면, 기존 `shareId`는 즉시 무효화한다.
- 공유 링크 접근 시 응답:
  - 삭제됨/권한 없음: `404`로 통일

## POL-017 Rate Limit(초안)

MVP에서는 간단한 제한만으로도 악성/실수 트래픽을 방어한다.

- POI 조회(nearby/along-route): IP 기준 60 req/min (초안)
- 코스 생성(업로드/라이딩 기반): userId 기준 10 req/day (초안)
- 공유 링크 발급: userId 기준 30 req/day (초안)
- 댓글 작성: userId 기준 30 req/day (초안)
- 댓글 신고: userId 기준 50 req/day (초안)

메모:

- 정확한 수치는 운영 지표를 보면서 조정한다.
- 구현은 in-memory(단일 인스턴스)로 시작하고 필요 시 Redis로 이동한다.

## POL-018 에러 코드 카탈로그(초안)

원칙(MVP):

- 응답 포맷은 `ApiResponse{ code(int), message, data }`를 기본으로 한다.
- `code`는 우선 HTTP status(예: 200/400/404/500)를 그대로 사용한다.
- 상세 분류(예: COURSE-404 같은 문자열 에러 ID)는 MVP에서는 "문서/로그"에서만 관리하고, 필요해지면 vNext에 `errorId` 필드를 추가한다.

문서/로그용 에러 ID(예시, 확장 가능):

- COURSE-404: 코스를 찾을 수 없음
- COURSE-403: 코스 접근 권한 없음(visibility)
- COURSE-400: 코스 생성 요청이 유효하지 않음(path 좌표 부족 등)
- POI-500: POI 동기화 실패(운영용)
- POI-400: 좌표/반경 파라미터 오류

## POL-019 코스 댓글(익명/노출/삭제)

익명 원칙:

- 코스 댓글은 UI에서 항상 작성자 표시를 `익명`으로 한다.
- API 응답에서도 작성자의 userId/email/카카오 프로필을 노출하지 않는다.
- 클라이언트는 `isMine` 같은 플래그로만 "내 댓글"을 구분한다.

## POL-020 여행 컬렉션 공개/편집 정책

컬렉션은 여러 코스를 여행 계획 단위로 묶는 사용자 콘텐츠다.

- 공개 범위:
  - `private`: 소유자만 조회 가능
  - `unlisted`: 링크를 아는 사람만 조회 가능(Phase 1 상세 공유는 후속)
  - `public`: 누구나 목록/상세 조회 가능
- 편집:
  - Phase 1에서는 소유자만 수정/코스 추가 가능
  - 공동 편집자는 vNext로 미룬다
- 중복:
  - 동일 컬렉션에는 같은 코스를 중복 추가할 수 없다
- 순서:
  - `positionIndex`로 여행 순서를 표현한다
  - 미지정 시 맨 뒤에 자동 추가한다

운영 메모:

- 공개 컬렉션은 향후 큐레이션/추천 피드의 입력 데이터가 될 수 있다.
- private 컬렉션 내용은 목록/상세 어디에서도 타 사용자에게 노출하지 않는다.
- 컬렉션이 공개여도 내부 코스가 비공개면 해당 코스는 상세에서 비노출 또는 제외 처리한다.

## POL-021 코스 하이라이트 작성/노출 정책

하이라이트는 코스 위 특정 좌표에 남기는 현장 포인트 메모다.

- 유형:
  - `viewpoint`, `restroom`, `water`, `cafe`, `danger`, `photo`, `note`
- 공개 범위:
  - `public`: 다른 사용자에게도 노출
  - `private`: 작성자 본인에게만 노출
- 작성 조건:
  - 로그인 사용자만 작성 가능
  - 좌표(lat/lon)와 type은 필수다
  - title/note 중 하나는 반드시 있어야 한다
- 노출:
  - 일반 조회는 공개 하이라이트만 보여준다
  - 작성자는 본인 private 하이라이트를 함께 본다

운영 메모:

- 신고/숨김/임시 만료는 vNext 운영 기능으로 추가한다.
- Phase 1에서는 `hidden=false` 기본값으로 생성하고, 운영 숨김 API는 아직 제공하지 않는다.

삭제 정책:

- 작성자만 삭제 가능
- 삭제는 soft delete(status=deleted) 권장
- 수정(edit)은 MVP2에서 제공하지 않는다(삭제+재작성)

## POL-022 댓글 모더레이션/신고(최소)

- 사용자는 댓글을 신고할 수 있다.
- 운영은 신고된 댓글을 숨김 처리할 수 있다(status=hidden).
- 숨김/삭제 댓글은 기본 조회에서 제외한다(운영 화면에서만 확인).

## POL-023 코스 모임(참여/정원) (MVP3)

- 모임 생성/참여/탈퇴는 인증 사용자만 가능하다.
- 정원(capacity)이 설정된 모임은 정원 초과 참여를 허용하지 않는다.
- 같은 유저의 중복 참여는 허용하지 않는다(unique meetup_id,user_id).
- 참여 동시성은 DB row-level lock으로 처리한다.
- 모임장은 leave할 수 없다(필요 시 모임 종료/취소를 사용).
- 기본 검증: `capacity`는 1~100, `startAt`은 과거 시각 불가.

## POL-024 Highlight / Crew Notes 신뢰도 정책

- Highlight는 `draft | visible | hidden | expired` 상태를 가진다.
- user-generated highlight는 신고 누적/운영 검토로 hidden 처리 가능해야 한다.
- 위험/공사/임시 정보는 `validUntil`을 권장한다.
- 같은 위치/같은 타입의 중복 Highlight는 병합 후보로 본다.

## POL-025 Safety Live Share 프라이버시 정책

- Live Share는 기본적으로 비공개이며, 만료형 링크로만 공유한다.
- 기본 만료 시간(초안): 생성 후 8시간 또는 라이딩 종료 시 즉시 종료
- 수신자는 실시간 위치/최근 갱신/예상 도착 정도만 본다.
- 공개 프로필/검색/인덱싱 대상이 되어서는 안 된다.

## POL-026 Route Package 저장 정책

- 오프라인 패키지는 기기 로컬 저장을 기본으로 한다.
- 패키지에는 원본 전체 개인정보를 넣지 않고, 공유용 패키지는 privacy clipping 정책을 따른다.
- stale package는 마지막 동기화 시각을 명시해야 한다.

## POL-027 Challenge/완주 판정 정책

- 초기 Challenge는 속도 경쟁보다 `완주`, `방문`, `탐방` 중심으로 설계한다.
- 동일 사용자의 중복 완료 집계 조건을 문서로 고정한다.
- 부정행위 방지보다 단순하고 설명 가능한 규칙을 우선한다.

## POL-028 MVP1 핵심 이벤트/지표 운영

- MVP1 핵심 이벤트 이름은 실행 SSOT와 동일하게 유지한다:
  - `home_viewed`
  - `featured_course_clicked`
  - `course_detail_viewed`
  - `guide_started`
  - `guide_off_route_detected`
  - `riding_save_requested`
  - `riding_save_succeeded`
  - `course_created_from_riding`
  - `course_share_issued`
  - `shared_course_opened`
- 제품 성공 판단은 최소 아래 4개 지표를 본다:
  - `course_detail_viewed`
  - `guide_started / course_detail_viewed`
  - `riding_save_succeeded / riding_save_requested`
  - `shared_course_opened`
- `shareId` 같은 민감 식별자는 외부 분석 도구에 raw 값으로 보내지 않는다.
