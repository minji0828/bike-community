# 프론트엔드 개발 가이드

- 문서 ID: FE-DEV-GUIDE
- 버전: v1.0
- 작성일: 2026-03-10
- 상태: 구현 계약 SSOT

## 0. 이 문서의 역할

이 문서는 현재 저장소의 프론트엔드 구현 기준 문서다.  
`frontend/`에서 작업할 때는 이 문서를 기본값으로 사용한다.

우선순위:

1. 이 문서
2. `설계/11_API_명세.md`
3. `설계/04_정책정의서.md`
4. `설계/adr/frontend/*.md`
5. 기존 RN/Expo 전제 문서

즉, 기존 문서에 RN/Expo 기준 설명이 남아 있어도 현재 저장소 프론트 구현 판단은 이 문서를 따른다.

## 1. 현재 타깃과 범위

- 현재 1차 프론트엔드는 `frontend/`의 Next.js App Router 웹 앱이다.
- 모바일 앱은 현재 저장소 기준 구현 대상이 아니다.
- 프론트 P0 핵심 흐름은 아래를 우선한다.
  - 홈: featured 코스 + 주변 화장실
  - 코스 상세: 코스 조회 + 공유 + 따라가기 진입
  - 가이드: L2 따라가기 UI
  - 라이딩 기록: GPS 추적 -> 로그인 상태에서 라이딩 저장 -> 코스 생성
  - 인증: 카카오 로그인 + `/api/auth/me` 재검증
- 프론트 P1 확장 흐름은 아래다.
  - 컬렉션: 생성/추가/상세 조회
  - 하이라이트: 조회/생성
  - 코스모임: 생성/참가/탈퇴
  - 모임 채팅: 참가자 전용 실시간 채팅

## 2. 기술 스택

- 프레임워크: Next.js 16 App Router
- UI 런타임: React 19
- 언어: TypeScript
- 스타일링: Tailwind CSS 4 + `components/ui` 프리미티브
- 지도: Kakao Maps JavaScript SDK
- 인증: Kakao 로그인 시작 + backend JWT 사용
- 실시간: native WebSocket + 경량 STOMP 래퍼

관련 ADR:

- `설계/adr/frontend/0001-nextjs-web-app-router-primary-frontend.md`
- `설계/adr/frontend/0002-kakao-auth-via-next-routes-and-backend-jwt.md`
- `설계/adr/frontend/0003-api-wrapper-and-local-state-first.md`
- `설계/adr/frontend/0004-kakao-maps-js-and-lightweight-stomp.md`
- `설계/adr/frontend/0005-course-core-flow-share-and-browser-gps-guide.md`

## 3. 디렉터리 책임

### 3.1 `frontend/app`

- 페이지 라우트와 Next route handler만 둔다.
- 페이지는 화면 오케스트레이션을 담당한다.
- 브라우저 API, 지도, 채팅, 로컬 UI 상태가 필요하면 client component로 둔다.

### 3.2 `frontend/components`

- 페이지에서 재사용하는 기능 컴포넌트를 둔다.
- `components/ui`는 공통 UI primitive다.
- 도메인 기능은 `auth`, `chat`, `collection`, `highlight`처럼 폴더로 분리한다.

### 3.3 `frontend/lib`

- 백엔드 API 호출 함수
- 타입/adapter
- 브라우저 저장소 유틸
- 지리 계산, PKCE, device UUID, STOMP 같은 기술 유틸

규칙:

- 페이지에서 raw `fetch`를 직접 쓰지 않는다.
- route handler는 예외적으로 backend proxy를 위해 raw `fetch`를 사용할 수 있다.

### 3.4 `frontend/hooks`

- 범용 훅만 둔다.
- 페이지 전용 훅이 필요하면 먼저 해당 페이지/기능 폴더 안에서 시작하고, 반복되면 `hooks/`로 올린다.

## 4. 라우트 맵

| Route | 역할 | 주요 API/통신 | 인증 |
|---|---|---|---|
| `/` | featured 코스, 주변 화장실 홈 | `GET /api/v1/courses/featured`, `GET /api/v1/pois/nearby` | 선택 |
| `/explore` | 탐색 실험 화면 | 현재 샘플 데이터 중심, 추후 탐색 API 연동 | 선택 |
| `/ride` | 라이딩 기록, 저장, 코스 생성 | `POST /api/v1/ridings`, `POST /api/v1/courses/from-riding` | 기록은 선택, 저장/생성은 필수 |
| `/course/[id]` | 코스 상세, 하이라이트, 코스모임, 컬렉션 추가 | `GET /api/v1/courses/{id}`, highlights, meetups, collections | 읽기 공개, 쓰기 인증 |
| `/course/[id]/guide` | L2 가이드 UI | `GET /api/v1/courses/{id}` | 선택 |
| `/share/[shareId]` | 공유 코스 공개 진입 페이지(MVP1 타깃 라우트) | `GET /api/v1/courses/public/{shareId}` | 없음 |
| `/profile` | 로그인/로그아웃, 인증 상태 표시 | `/auth/kakao/*`, `/api/auth/me` | 인증 상태 확인 |
| `/collections/[collectionId]` | 컬렉션 상세 | `GET /api/v1/collections/{id}` | 선택 |
| `/meetups/[meetupId]/chat` | 모임 참가자 채팅 | `GET /api/v1/meetups/{id}`, `ws-stomp` | 필수 |
| `/api/auth/me` | same-origin auth proxy | backend `/api/v1/auth/me` | 내부 route |
| `/api/health` | 웹 앱 health check | local response | 없음 |

## 5. 데이터 연동 규칙

### 5.1 공통 fetch 규칙

- 모든 백엔드 호출은 `frontend/lib/api.ts`의 `apiFetch` 또는 `appRouteFetch`를 사용한다.
- 백엔드 응답 기본 계약은 `ApiResponse<T>` envelope이다.
- 공통 타임아웃, 에러 파싱, 토큰 헤더 주입은 fetch wrapper가 담당한다.

### 5.2 `apiFetch`를 쓰는 경우

- 일반적인 backend direct 호출
- 공개 API
- Bearer token을 헤더로 직접 실어도 되는 보호 API

예:

- 코스 상세
- featured 코스
- 하이라이트 조회/생성
- 모임 생성/참가/탈퇴
- 라이딩 저장

### 5.3 `appRouteFetch`를 쓰는 경우

- same-origin route를 통해 backend를 프록시해야 할 때
- cookie 기반 인증 확인처럼 브라우저와 서버 경계가 필요한 요청

현재 기본 예:

- `/api/auth/me`

규칙:

- 단순 backend passthrough가 필요하더라도 먼저 `apiFetch`로 충분한지 검토한다.
- app route를 새로 만들면 "왜 frontend 직접 호출이 아닌지"를 코드와 문서에 남긴다.

### 5.4 도메인별 API 파일 규칙

- `lib/courses.ts`: 코스 조회/adapter
- `lib/pois.ts`: POI 조회
- `lib/ridings.ts`: 라이딩 저장, 코스 생성
- `lib/collections.ts`: 컬렉션 API
- `lib/highlights.ts`: 하이라이트 API
- `lib/meetups.ts`: 모임 REST API
- `lib/stomp.ts`: 채팅 transport 유틸

새 도메인 API는 같은 패턴으로 `lib/<domain>.ts`에 추가한다.

## 6. 인증 규칙

### 6.1 기본 정책

- 프론트가 저장하는 토큰은 backend service JWT다.
- Kakao provider token을 일반 API 호출 토큰으로 쓰지 않는다.
- 인증 상태의 단일 진입점은 `AuthProvider`다.

### 6.2 로그인 플로우

1. `/auth/kakao/start`에서 PKCE/state/nonce를 준비한다.
2. 카카오 authorize로 리다이렉트한다.
3. `/auth/kakao/callback`이 code/state를 받는다.
4. `/auth/kakao/finalize`가 backend `/api/v1/auth/kakao`와 통신한다.
5. backend가 발급한 service JWT를 저장한다.
6. `AuthProvider`가 `/api/auth/me`로 재검증한다.

### 6.3 저장소 규칙

- 토큰/로그인 세션 저장은 `lib/auth.ts`만 담당한다.
- 다른 페이지나 컴포넌트에서 localStorage/cookie/sessionStorage를 직접 만지지 않는다.
- 로그아웃은 `AuthProvider.logout()`만 사용한다.

### 6.4 보호 API 사용 규칙

- 보호 API는 `useAuth()`에서 받은 `token`을 `apiFetch`에 넘긴다.
- token이 없으면 UI에서 먼저 차단하고, backend 실패 메시지도 함께 처리한다.
- 라이딩 저장, 코스 생성, 공유 링크 발급은 모두 보호 API로 본다.

## 7. 좌표, 지도, 위치 규칙

### 7.1 좌표 이름 규칙

- 프론트 내부 지도/뷰 모델: `{ lat, lng }`
- API DTO: `{ lat, lon }`
- 변환은 API boundary에서만 한다.

예:

- backend course detail 응답의 `point.lon`은 프론트 `routePath.lng`로 바꿔 저장한다.
- 라이딩 저장 payload는 프론트 `lng`를 API `lon`으로 바꿔 전송한다.

### 7.2 지도 SDK 규칙

- 카카오 지도 SDK 로딩은 `components/map-view.tsx`만 담당한다.
- 다른 컴포넌트에서 script tag를 직접 삽입하지 않는다.
- 새 지도 기능도 먼저 `MapView` 확장으로 해결 가능한지 본다.

### 7.3 위치 권한과 GPS 규칙

- 홈과 라이딩은 브라우저 Geolocation API를 사용한다.
- 위치 실패는 graceful degradation으로 처리한다.
- 라이딩 기록 샘플링/노이즈 정책은 `설계/23_라이딩_GPS_추적_기록_설계.md`를 따른다.

### 7.4 L2 가이드 규칙

- L2 진행률과 오프루트 감지는 클라이언트가 계산한다.
- 서버는 코스 path, warnings, metadata를 제공한다.
- 음성 내비, turn-by-turn, 재탐색은 현재 범위 밖이다.

## 8. 상태관리와 렌더링 규칙

### 8.1 상태관리 기본값

- 공유 전역 상태는 `AuthProvider`만 사용한다.
- 페이지 로딩/에러/선택/폼 상태는 페이지 또는 기능 컴포넌트 로컬 state로 둔다.
- Redux, Zustand, TanStack Query 같은 별도 상태관리 도구는 새 ADR 없이 도입하지 않는다.

### 8.2 렌더링 기본값

- 브라우저 API가 없는 정적 화면은 server component를 우선 검토한다.
- 아래는 client component가 기본이다.
  - geolocation
  - 지도 렌더링
  - WebSocket
  - auth context 사용
  - 복잡한 폼/상호작용

### 8.3 sample-data 사용 규칙

- `sample-data`는 임시 fallback이나 UI 뼈대 확인용이다.
- 새 기능을 만들 때 기본값으로 sample-data fallback을 늘리지 않는다.
- 실제 API가 준비되면 sample fallback은 제거 또는 최소화한다.

## 9. 모임/채팅 규칙

### 9.1 REST와 WebSocket 경계

- 모임 생성/목록/상세/참가/탈퇴는 REST다.
- 채팅만 WebSocket/STOMP를 사용한다.

### 9.2 현재 채팅 채널 규칙

- WebSocket endpoint: `/ws-stomp`
- 구독 topic: `/topic/meetups/{meetupId}/chat`
- 개인 history queue: `/user/queue/meetups/{meetupId}/chat.history`
- 송신 destination:
  - `/app/meetups/{meetupId}/chat.history`
  - `/app/meetups/{meetupId}/chat.send`

### 9.3 채팅 구현 규칙

- STOMP 연결은 `SimpleStompClient`를 사용한다.
- 채팅 페이지 외부에서 임의로 소켓 연결을 만들지 않는다.
- 재연결 정책, offline queue, 읽음 상태가 필요해지면 새 ADR을 만든다.

## 10. UI와 컴포넌트 규칙

- 공통 UI는 `components/ui`를 우선 사용한다.
- 기능 컴포넌트는 페이지에서 직접 JSX를 복붙하지 말고 `components/<feature>`로 분리한다.
- 모바일 우선 레이아웃(`max-w-md`)을 기본으로 유지한다.
- UI primitive에 도메인 로직을 넣지 않는다.

## 11. 환경 변수

필수 또는 사실상 필수:

- `NEXT_PUBLIC_API_BASE_URL`
- `NEXT_PUBLIC_KAKAO_CLIENT_ID`
- `NEXT_PUBLIC_KAKAO_MAP_JS_KEY`

상황별 선택:

- `NEXT_PUBLIC_KAKAO_REDIRECT_URI`

규칙:

- 브라우저에서 써야 하는 값만 `NEXT_PUBLIC_` prefix를 쓴다.
- 비밀값은 frontend에 넣지 않는다.

## 12. 새 기능 추가 기본 절차

1. 관련 API/정책/스키마 문서를 먼저 확인한다.
2. 기술 선택이 새로 필요하면 frontend ADR을 먼저 만든다.
3. `lib/<domain>.ts`에 타입과 fetch 함수를 만든다.
4. 기능 컴포넌트를 `components/<feature>`에 만든다.
5. 페이지에서 오케스트레이션한다.
6. API shape가 바뀌면 `설계/11_API_명세.md`를 갱신한다.

## 13. 프론트 작업 Definition of Done

- route 책임이 기존 구조와 충돌하지 않는다.
- API 호출이 `lib/api.ts` 규칙을 따른다.
- 인증이 필요한 액션은 token 유무와 실패 메시지를 처리한다.
- 좌표 변환이 `lon <-> lng` 규칙을 지킨다.
- 지도/채팅은 기존 진입점(`MapView`, `SimpleStompClient`)을 재사용한다.
- 기술 선택이 바뀌면 frontend ADR을 갱신한다.
