# MVP 요구사항 정의서 (가칭: 가자)

- 문서 ID: MVP-REQ
- 버전: v0.4
- 작성일: 2026-02-25
- 상태: 초안(코스 메타데이터 v1 반영)

이 문서는 `설계/00_문제정의_가설_가치검증.md`의 문제/가치 가설을 전제로 MVP 범위를 고정합니다.

## 0. 한 줄 소개

"가벼운 라이딩부터 자전거 여행까지"를 돕는 지도 앱. 내가 가는 길을 기록해 코스로 만들고, 코스/경로 주변의 필요한 POI(우선: 공공화장실)를 빠르게 보여주고 공유한다. MVP는 코스를 따라가는 가이드(진행률/이탈)까지 제공한다. (카페/편의점 등 확장은 vNext)

## 1. 문제 정의

- 라이딩 중 필요한 시설(우선: 화장실)을 "현재 위치"만으로 찾으면, 실제 이동 경로와 무관한 정보가 섞인다.
- 사용자가 자주 가는 길/좋았던 길을 코스로 저장/공유하기 어렵다.
- 트래픽이 증가하면 지도/코스 조회가 폭증(read-heavy)하고, 경로 주변 검색(geo query)이 비용이 큰 병목이 된다.

## 2. 목표(Goals)

- G1: "내 주변"과 "내 경로 주변"의 POI를 지도에서 즉시 확인한다.
- G2: 라이딩 경로를 기록해 코스화하고, 코스를 링크로 공유/재사용한다.
- G3: 성장 시 병목이 되는 조회/검색을 캐시/프리컴퓨트로 흡수 가능한 구조로 시작한다.
- G4: 코스 따라가기 가이드(레벨 2)를 제공해 실제 사용성을 검증한다.
- G5: 코스 콜드스타트를 위해 기본 제공 코스(큐레이션)를 준비한다.
- G6: 산책형 코스와 여행형 대표 코스를 모두 담을 수 있는 코스 플랫폼의 기반을 만든다.

## 3. 이번 MVP에서 제외(Non-Goals)

- 팔로우/피드/댓글 등 커뮤니티 풀세트
- 사용자 프로필 기반 개인화/학습형 추천(ML)
- LLM이 "새 코스"를 생성하는 라우팅(도로망 기반 코스 생성)
- 결제/구독
- 오프라인 지도(다운로드)
- 턴바이턴 내비(음성/교차로 안내/재탐색) = 레벨 4는 장기 목표로 별도 단계
- 실시간 다자간 위치 공유

## 4. 플랫폼/구성

- 클라이언트: React Native (지도 화면 + 기록 + 코스 공유)
- 서버: Spring Boot + Postgres(PostGIS)
- POI 데이터 소스: 외부 공공 데이터(동기화 후 DB 서빙)

### 길안내(내비게이션) 레벨 정의

- 레벨 2 (MVP): 코스 Polyline + 내 위치 + 진행률/남은 거리 + 이탈(off-route) 감지
  - 포함: 코스 선 렌더링, 재센터링, 간단 안내 UI
  - 제외: 교차로별 지시, 음성 안내, 재탐색(리라우팅)
- 레벨 4 (장기): 턴바이턴(TBT) 내비게이션
  - 포함: maneuver 기반 안내(좌/우/직진), 음성, 이탈 시 재탐색

## 5. 타겟 사용자/핵심 시나리오

- P1: 주말 산책 라이더: "그냥 켜서" 주변 화장실을 보며 편하게 탄다.
- P2: 코스 수집가: 마음에 드는 길을 코스로 저장하고, 친구에게 링크로 공유한다.
- P3: 여행/탐방형 라이더: 대표 코스나 GPX 코스를 따라가며 필요한 편의정보와 공유/동행 경험을 원한다.

핵심 시나리오(상위):

- S0: 추천/기본 코스 보기 -> 코스 상세 -> 코스 따라가기
- S1: 앱 실행 -> 지도에서 내 위치 + 주변 POI 표시
- S2: 라이딩 시작 -> 경로 기록 -> 종료 -> 코스 저장
- S3: 코스 상세 -> 경로 주변 POI 확인 -> 코스 공유(링크)
- S4: 코스 열기 -> 코스 따라가기 시작 -> 진행률/이탈 안내
- S5: 여행형 Featured 코스 보기 -> 코스 공유 또는 모임/채팅(후속 단계)

## 6. MVP 범위 (Must Have)

아래 범위는 "코스가 콘텐츠가 된다"를 검증하기 위한 최소 단위다.

### A. 지도/POI

- A1: 지도 화면에서 현재 위치 기준 반경 POI(화장실) 표시
- A2: 코스(경로) 기준 경로 주변 POI 표시(along-route)
- A3: POI 동기화(전체/증분) 및 실패 시 재시도/로그

### B. 기록/코스

- B1: 라이딩 기록(경로 좌표) 수집 및 서버 저장
- B2: 코스 생성(제목/설명/태그 최소) 및 코스 목록/상세 조회
- B3: 코스 공유 링크(공개 코스 조회) + 조회수 같은 최소 지표
- B4: 기본 제공 코스(큐레이션) 세트 준비 + 노출(리스트/상세)
- B5: 코스 메타데이터 v1 저장/노출 + 자동 파생치(거리/예상시간/POI 요약) 생성

메모:

- Featured 코스는 산책형 루프만이 아니라, 여행/관광형 대표 코스를 포함할 수 있다.
- 다만 MVP 우선순위는 여전히 "쉽게 열고 따라가며 필요한 편의시설을 확인하는 경험"에 둔다.

### C. 트래픽/성능 최소 대응(초기 설계)

- C1: 코스 상세/경로 주변 POI 결과를 캐시(TTL)할 수 있는 키 설계
- C2: 외부 POI API 호출을 런타임 요청에서 제거(동기화 후 DB 서빙)
- C3: 요청 폭주에 대비한 기본 rate-limit 대상 정의(코스 생성/경로 주변 검색)

### D. 길안내(코스 따라가기, 레벨 2)

- D1: 코스 Polyline 렌더링 + 현재 위치 표시 + 코스 재센터링
- D2: 진행률(%) / 남은 거리 표시(코스 geometry 기반)
- D3: 코스 이탈 감지(거리 임계값) + 사용자 피드백(배너/진동 등) + 복귀 유도
- D4: GPS 노이즈를 고려한 "이탈/복귀" 상태 전환(히스테리시스) 설계

### E. 코스 메타데이터 v1(추천/검색 공통)

목표: 추천/필터/검색이 가능한 수준으로 코스를 "설명 가능한 데이터"로 만들되, 사용자 입력 부담은 최소화한다.

원칙:

- 사용자는 최소 입력만 한다(제목/태그/특이사항 선택).
- 나머지 대부분은 서버가 코스 geometry + POI DB로 자동 산출한다.
- LLM(에이전트)은 "자연어 -> 조건" 변환까지만 담당하고, 추천/랭킹은 결정적(규칙 기반)으로 구현한다.

#### E1. 코스 메타데이터 그룹

- Core(필수): 코스 기본 속성/접근 제어/지도 렌더링에 필요한 값
- Derived(자동): 코스 geometry/POI를 기반으로 계산되는 값(추천/요약)
- Annotation(선택 입력): 태그/특이사항/하이라이트 등 사람이 추가하는 값
- Ops(운영): 검증/피처드/인기도 같은 운영용 값

#### E2. 메타데이터 필드(초안)

Core

- `title` (필수)
- `description` (선택)
- `visibility`: `public | unlisted | private`
- `sourceType`: `curated | ugc | import`
- `path`: LineString (SRID 4326, 좌표순서 lon,lat)
- `region` / `startArea`: 추천을 위한 대략 위치 정보(정확한 집/출발지 노출 방지)

Derived (서버 자동 산출)

- `distanceKm`
- `loop`: 시작/끝이 가까우면 true(임계값 정책)
- `estimatedDurationMin`: 기본 속도 가정 + 난이도 보정(초기에는 단순 규칙)
- `bbox`: 지도 뷰포트용 bounding box
- `amenitiesSummary` (코스 주변 n미터 버퍼 기준)
  - `toiletCount`
  - `cafeCount` (데이터 소스가 준비되면)
  - (선택) `toiletMaxGapKm` / `toiletAvgGapKm` (코스 거리 기준, 후순위)

Annotation (사용자/큐레이터 입력)

- `tags[]`: 멀티 선택(예: 강변/한적/초보/가족/야경/업힐)
- `notes`: 짧은 코스 소개/추천 이유
- `warnings[]`: 코스 특이사항/주의 구간(구조화)
- `highlights[]`: 볼거리/정차 추천 지점(구조화)

Ops (운영)

- `verifiedStatus`: `unverified | community | curated`
- `lastVerifiedAt`
- `featuredRank`(선택): 기본 제공 코스/추천 리스트 노출 순서
- `popularity`(선택): 조회/저장/따라가기 수 집계

#### E3. 특이사항(경고) 구조(권장)

"특이사항"은 프리텍스트 한 줄보다, 추천/필터/표시에 활용할 수 있도록 구조화한다.

- `type`(enum 예시): `busyCrossing | steepDescent | gravel | narrowLane | construction | noLight`
- `severity`: 1~3
- `at`: lon/lat (선택)
- `radiusM`: 영향 반경(선택)
- `note`: 사용자에게 보여줄 1줄 설명(선택)
- `validUntil`: 공사/임시 이슈 만료(선택)

MVP 정책: 경고는 "참고" 정보로 제공하며, 완전성을 보장하지 않는다(면책/안내 문구는 정책 문서로 확정).

#### E4. 메타데이터 생성/갱신 책임

- [Client] 입력: `title`, `tags[]`, `notes`, `warnings[]` (선택)
- [Server] 자동: `distanceKm`, `bbox`, `loop`, `estimatedDurationMin`, `amenitiesSummary`
- [Backoffice] 운영: `featuredRank`, `verifiedStatus`, 태그 사전(관리)

#### E5. 메타데이터 재계산 트리거(초안)

- 코스 생성/업로드 시: 즉시(동기)로 `distanceKm/bbox/loop` 산출
- 코스 생성/업로드 후: 비동기(워커/배치)로 `amenitiesSummary` 산출
- POI 동기화 후: "전체 재계산"은 비용이 크므로, 우선 featured/최근 코스부터 재계산

### E6. 상품성 강화 후보(vNext)

- `Trip Collection`: 코스 여러 개를 여행/주말 일정으로 묶는 상위 콘텐츠
- `Highlights & Crew Notes`: 코스 위 포인트/팁/주의사항을 좌표 기반으로 기록
- `Safety Live Share`: 실시간 위치/ETA/최근 상태를 만료형 링크로 공유
- `Route Package`: geometry + 핵심 POI + warnings를 오프라인 대비 패키지로 저장
- `Collection Complete / Challenge`: 완주/탐방 중심 리텐션 기능

메모:
- 상세 우선순위와 이유는 `설계/25_기능확장_성장전략.md`를 따른다.
- 시장 벤치마크는 `docs/travel-cycling-product-benchmark-2026-03-07.md`를 참조한다.

## 7. Out of Scope (이번 MVP에서 제외)

- 회원가입/로그인 완성형(토큰/리프레시/소셜 로그인)
- 신고/차단/모더레이션
- 코스 검색/정렬/랭킹(인기 코스)
- 자연어 기반 추천 에이전트(LLM UI)
- Trip Collection / Highlight / Live Share / Challenge의 완성형 기능
- 코스 편집 히스토리/버저닝
- 이미지/썸네일 자동 생성(추후 워커)
- 관리자 백오피스 UI(대신 관리용 API만 우선)
- 턴바이턴 내비(레벨 4): 음성 안내, 교차로 안내, 재탐색, 도로망 기반 라우팅

## 8. 사용자 스토리 + 수용 기준(AC)

### US-001 주변 POI 조회(화장실)

- As a 사용자, 지도에서 내 주변 화장실을 보고 싶다.
- AC:
  - 기본 반경이 적용된다(예: 500m)
  - POI 카테고리 필터는 MVP에서는 화장실만 제공하되, 추후 확장 가능한 구조를 유지한다
  - 응답은 거리 또는 기본 정렬 기준이 일관적이다

### US-002 경로 주변 POI 조회(화장실)

- As a 사용자, 내가 갈/간 경로 주변의 화장실을 보고 싶다.
- AC:
  - 경로 좌표가 비어있으면 빈 리스트를 반환한다
  - 좌표 순서(lon,lat) 규칙이 문서화되고 검증된다
  - 동일 코스/동일 반경 요청은 캐시로 빠르게 응답 가능하다(설계 요건)

### US-003 라이딩 기록 저장

- As a 사용자, 라이딩을 기록하고 코스로 저장하고 싶다.
- AC:
  - 최소 2개 이상 좌표가 없으면 저장 정책이 명확하다(거부/임시 저장)
  - 저장 후 코스 상세에서 경로를 재표시할 수 있다

### US-004 코스 공유

- As a 사용자, 코스를 링크로 공유하고 싶다.
- AC:
  - 링크로 공개 코스 상세를 볼 수 있다(앱/웹 중 MVP 정책 선택)
  - 공유 시 개인정보(정확한 집/출발지) 보호 옵션은 추후로 남기되, 최소한 비공개 코스는 노출되지 않는다

### US-005 코스 따라가기 가이드(레벨 2)

- As a 사용자, 저장된 코스를 열고 내 위치가 코스에서 어디쯤인지 보며 따라가고 싶다.
- AC:
  - 코스 선과 내 위치가 지도에 동시에 표시된다
  - 진행률(%) 또는 남은 거리 중 최소 1개가 표시된다
  - 코스에서 일정 거리 이상 벗어나면 "이탈"로 표시되고, 다시 가까워지면 "복귀"로 전환된다
  - 턴바이턴(좌/우/직진 지시, 음성, 재탐색)은 제공하지 않는다

## 9. 성공 지표(Success Metrics)

- M1: 코스 저장 성공률(저장 요청 대비 성공) 99%+
- M2: 코스 상세 조회 p95 < 500ms (단일 인스턴스 기준, 캐시 적용 전/후 비교)
- M3: 경로 주변 POI 조회 p95 < 800ms (초기 목표, 이후 개선)
- M4: 외부 POI 동기화 실패율/재시도 성공률을 로그로 관측 가능
- M5: 코스 따라가기 화면에서 이탈/복귀 표시가 과도하게 흔들리지 않는다(히스테리시스 적용)

## 10. 릴리즈 컷(마일스톤)

- v0.1: POI 동기화 + 주변/경로주변 조회 API + Swagger
- v0.2: 라이딩 저장 + 코스 목록/상세 + 공유 링크(공개 조회) + 기본 캐시 키 설계
- v0.3: rate-limit, 코스 프리컴퓨트(워커/배치), 백오피스 최소 API
- v1.0: 턴바이턴(레벨 4) 설계 확정 + 구현(외부/내장/자체 방식 중 선택)

## 11. 리스크/가정/대응

- R1 외부 POI API 변동(스키마/쿼터): 동기화 계층을 분리하고 DB 서빙으로 보호
- R2 위치/경로는 개인정보: 운영 로그에서 위경도 원본을 최소화, retention 정책은 별도 문서에서 확정
- R3 read-heavy 트래픽: 코스 상세/along-route 결과 캐시, POI는 공간 인덱스 기반 조회
- R4 경로 데이터 크기: 경로 단순화/샘플링 전략을 추후 도입(저장 포맷/버전 고려)
- R5 GPS 노이즈: 이탈 오탐/진동 문제 -> 임계값+히스테리시스, 샘플링/스무딩으로 완화
