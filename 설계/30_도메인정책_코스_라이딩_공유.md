## 1. 라이딩 생성 정책

- 라이딩 저장은 로그인 사용자만 수행할 수 있다.
- 라이딩 소유권은 요청 바디가 아니라 서버가 해석한 JWT `sub`에서 결정한다.
- `deviceUuid`는 설치 단위 식별과 중복/운영 추적 보조 값으로 유지하되, 권한 판단의 기준으로 사용하지 않는다.
- 라이딩 경로는 최소 2개 좌표가 있어야 하며, 각 좌표는 유효한 위도/경도 범위를 만족해야 한다.
- 서버 내부 지오메트리 생성 시 좌표 순서는 항상 `(lon, lat)`다.
- 저장된 라이딩 경로는 민감 데이터로 취급하며, 운영 로그에 원본 좌표를 직접 남기지 않는다.

### 1-1. 라이딩 생성 정책 코드

| 정책코드 | 항목 | 정책 정의 | 현재 구현 기준 |
|---|---|---|---|
| RID-P-001 | 인증 | 라이딩 저장은 인증된 사용자만 가능하다. | 구현됨 |
| RID-P-002 | 소유권 | 라이딩 소유자는 JWT `sub`로 확정한다. 요청 바디 `userId`는 사용하지 않는다. | 구현됨 |
| RID-P-003 | 디바이스 식별 | `deviceUuid`는 필수 입력이지만 권한 판단 값은 아니다. | 구현됨 |
| RID-P-004 | 경로 최소 조건 | `path`는 최소 2개 좌표가 필요하다. | 구현됨 |
| RID-P-005 | 좌표 유효성 | 위도 `-90~90`, 경도 `-180~180` 범위를 벗어나면 실패한다. | 구현됨 |
| RID-P-006 | 지오메트리 표준 | 서버/JTS/PostGIS 좌표 생성은 `(lon, lat)` 순서를 사용한다. | 구현됨 |
| RID-P-007 | 민감정보 보호 | 라이딩 원본 좌표는 운영 로그에 직접 남기지 않는다. | 정책 기준 |

## 2. 코스 생성 정책

- 코스 생성은 수기 입력, GPX 업로드, 라이딩 기반 생성 세 가지 경로를 허용한다.
- 세 경로 모두 인증된 사용자만 수행할 수 있다.
- 새 코스의 소유권은 인증 사용자에게 귀속된다.
- 코스 기본 공개 범위는 `private`다.
- `sourceType` 미입력 시 기본값은 `ugc`다.
- `curated`로 생성된 코스만 자동으로 `verifiedStatus=curated`를 가진다. 그 외는 `unverified`다.
- 코스 경로는 최소 2개 좌표가 있어야 하며, 파생 메타데이터는 서버가 계산한다.
- 태그는 정규화된 소문자 key로 관리하고, 없는 태그는 일반 태그로 자동 생성할 수 있다.

### 2-1. 코스 생성 정책 코드

| 정책코드 | 항목 | 정책 정의 | 현재 구현 기준 |
|---|---|---|---|
| CRS-P-001 | 인증 | 코스 생성은 인증된 사용자만 가능하다. | 구현됨 |
| CRS-P-002 | 소유권 | 코스 소유자는 JWT `sub`로 확정한다. | 구현됨 |
| CRS-P-003 | 기본 공개범위 | `visibility` 미입력 시 기본값은 `private`다. | 구현됨 |
| CRS-P-004 | sourceType 기본값 | `sourceType` 미입력 시 기본값은 `ugc`다. | 구현됨 |
| CRS-P-005 | verified 상태 | `sourceType=curated`일 때만 `verifiedStatus=curated`다. | 구현됨 |
| CRS-P-006 | 경로 최소 조건 | 코스 path는 최소 2개 좌표가 필요하다. | 구현됨 |
| CRS-P-007 | 파생 메타데이터 | 거리, bbox, loop, 예상 시간은 서버가 계산한다. | 구현됨 |
| CRS-P-008 | 태그 정규화 | 태그는 trim/lowercase/distinct 처리 후 저장한다. | 구현됨 |
| CRS-P-009 | 경고 저장 | 경고(warnings)는 코스 생성 시 함께 저장할 수 있다. | 구현됨 |

## 3. 라이딩 기반 코스 전환 정책

- 라이딩 기반 코스 생성은 로그인 사용자 자신의 라이딩에서만 가능하다.
- 대상 라이딩이 존재하지 않으면 실패한다.
- 라이딩에 저장된 path 좌표가 부족하면 코스 생성은 실패한다.
- `description`이 비어 있으면 `notes`를 설명 대체값으로 사용할 수 있다.
- 라이딩 기반 코스 생성 시 `deviceUuid`는 라이딩에서 가져오되, 권한 판단은 여전히 사용자 소유권으로 한다.

### 3-1. 라이딩 기반 코스 전환 정책 코드

| 정책코드 | 항목 | 정책 정의 | 현재 구현 기준 |
|---|---|---|---|
| CRS-P-010 | 원본 라이딩 존재 | `ridingId`는 반드시 존재해야 한다. | 구현됨 |
| CRS-P-011 | 라이딩 소유권 | 라이딩 소유자만 해당 라이딩으로 코스를 만들 수 있다. | 구현됨 |
| CRS-P-012 | 경로 품질 | 라이딩 path 좌표가 2개 미만이면 코스 생성은 실패한다. | 구현됨 |
| CRS-P-013 | 설명 대체 | `description` 미입력 시 `notes`를 설명 대체값으로 사용할 수 있다. | 구현됨 |
| CRS-P-014 | 디바이스 상속 | 라이딩에서 생성한 코스는 원본 `deviceUuid`를 유지할 수 있다. | 구현됨 |

## 4. 공유 링크 정책

- 공유 링크는 인증된 코스 소유자만 발급할 수 있다.
- 공유 링크 발급은 코스별 idempotent 하게 동작하며, 기존 `shareId`가 있으면 재사용한다.
- 공유 코스 읽기는 `public` 또는 `unlisted`만 허용한다.
- `private` 코스는 `shareId`가 있어도 공개 조회할 수 없다.
- 공유 링크가 무효하거나 접근할 수 없는 코스는 `404`로 통일한다.
- 공유 GPX 다운로드도 같은 공개 범위 정책을 따른다.

### 4-1. 공유 링크 정책 코드

| 정책코드 | 항목 | 정책 정의 | 현재 구현 기준 |
|---|---|---|---|
| SHR-P-001 | 발급 권한 | 공유 링크는 인증된 코스 소유자만 발급할 수 있다. | 구현됨 |
| SHR-P-002 | 소유자 필수 | `ownerUserId`가 없는 코스는 공유 링크를 발급할 수 없다. | 구현됨 |
| SHR-P-003 | 재발급 방식 | 기존 `shareId`가 있으면 재발급 시 재사용한다. | 구현됨 |
| SHR-P-004 | 공개 조회 범위 | 공유 코스 조회는 `public`, `unlisted`만 허용한다. | 구현됨 |
| SHR-P-005 | 비공개 차단 | `private` 코스는 `shareId`가 있어도 공개 조회할 수 없다. | 구현됨 |
| SHR-P-006 | 실패 응답 | 공유 대상이 없거나 접근 불가면 `404`로 응답한다. | 구현됨 |
| SHR-P-007 | 공유 GPX | 공유 GPX 조회도 동일한 공개 범위 정책을 따른다. | 구현됨 |

## 5. 코스 조회/피처드 정책

- 피처드 코스는 `featuredRank`가 있는 코스만 노출한다.
- 피처드 조회는 기본적으로 공개 읽기다.
- 코스 상세 조회는 현재 구현 기준으로 공개 읽기를 허용한다. 단, 공유 읽기와 비공개 보호 규칙은 별도로 적용한다.
- `amenitiesSummary`는 비어 있을 수 있으며, 클라이언트는 null 가능성을 허용해야 한다.

## 6. 운영/감사 정책

- 코스/라이딩/공유 도메인의 실패는 `BusinessException(code, message)`로 통일한다.
- 현재 MVP1은 DB 감사 로그를 별도 저장하지 않는다.
- 대신 정책 ID를 코드 주석, 테스트 이름, 운영 작업 일지에 남겨 추적성을 확보한다.
- 추후 관리자 화면이 도입되면 공유 링크 발급, 코스 비노출, GPX 다운로드 같은 핵심 행위는 감사 로그 대상으로 승격한다.

### 6-1. 운영 정책 코드

| 정책코드 | 항목 | 정책 정의 | 현재 구현 기준 |
|---|---|---|---|
| OPS-P-CRS-001 | 에러 규약 | 도메인 실패는 HTTP status + `ApiResponse.code/message`로 표현한다. | 구현됨 |
| OPS-P-CRS-002 | 정책 추적성 | 도메인 핵심 로직에는 정책 ID를 주석으로 연결한다. | 이번 작업에서 반영 |
