## 1. 기능 요구사항 (FR)

각 항목은 구현 전/후에 "수용 기준(AC)"을 반드시 채웁니다.

표기:

- `[Client]` 모바일 앱(React Native)
- `[Server]` 백엔드 API
- `[Ops]` 운영/관리 작업

### FR-001 현재 위치 표시 [Client]

- 설명: 앱에서 현재 위치를 지도의 기준으로 표시한다.
- AC:
  - 위치 권한이 없으면 권한 요청 또는 제한 모드로 전환한다
  - 정확도/권한 상태가 UI로 식별 가능하다(예: "정확도 낮음")

### FR-002 라이딩 경로 기록 [Client]

- 설명: 라이딩 중 위치를 주기적으로 수집해 경로를 만든다.
- AC:
  - 기록 시작/일시정지/종료가 가능하다
  - 기록 좌표가 너무 많아지면 단순화/샘플링으로 렌더링 성능을 유지한다

### FR-003 라이딩 저장 [Server]

- 설명: 라이딩 메타 + 경로(LineString)를 저장한다.
- 입력: userId(또는 deviceId), 좌표 리스트(lon,lat), 시간/거리 등 메타(선택)
- 출력: 저장된 라이딩(또는 코스 생성에 사용할 원본)
- AC:
  - 좌표가 2개 미만이면 정책에 따라 거부 또는 임시 저장한다
  - 좌표 순서(lon,lat) 규칙이 검증된다

### FR-004 코스 생성/관리 [Server]

- 설명: 라이딩 경로(또는 업로드한 경로)로 코스를 생성한다.
- AC:
  - 제목/설명/태그 입력 정책을 명확히 한다(최소 title 필수)
  - 코스 공개 범위(public/unlisted/private)를 설정할 수 있다(최소 public/unlisted)
  - 코스 sourceType(curated/ugc/import)를 구분할 수 있다
  - 코스 특이사항(경고)/하이라이트는 선택 입력으로 저장 가능하다(구조화 권장)

### FR-005 코스 목록/상세 조회 [Server]

- 설명: 내 코스 목록과 코스 상세(경로 포함)를 조회한다.
- AC:
  - 상세 응답에 코스 geometry(또는 polyline)와 요약(거리 등)이 포함된다
  - 공개 범위에 따라 접근 제어가 일관적이다

### FR-006 코스 공유 링크 [Server]

- 설명: 코스를 링크(또는 코드)로 공유한다.
- AC:
  - 공유 링크로 코스 상세를 조회할 수 있다(공개 범위 준수)
  - 조회수 같은 최소 지표를 집계할 수 있다(추후 이벤트 기반으로 분리 가능)

### FR-007 주변 POI 조회(반경) [Server]

- 설명: (lat, lon, radius[m], category) 기반으로 POI 반환
- AC:
  - radius 기본값 500m
  - category 필터 구조를 가진다(MVP는 화장실만 제공, vNext에서 카페/편의 확장)
  - 거리/정렬 기준(예: 가까운 순)이 일관적이다

### FR-008 경로 주변 POI 조회(along-route) [Server]

- 설명: 경로 좌표 리스트 + radius[m]로 경로 주변 POI 반환
- AC:
  - 빈 경로면 빈 리스트
  - 좌표 순서(lon,lat) 오류를 방지한다
  - 동일 코스/동일 반경 요청은 캐시 가능한 키로 식별된다

### FR-009 POI 동기화(전체/증분) [Server/Ops]

- 설명: 외부 API로부터 POI(우선: 화장실)를 적재/갱신한다. (카페/편의 확장은 vNext)
- AC:
  - 전체: 기존 데이터 삭제 후 새로 적재(또는 안전한 교체 전략)
  - 증분: externalId 기준 upsert/merge
  - 실패 시 제한된 재시도 + 운영 로그/알림(추후)

### FR-010 코스 따라가기 가이드(레벨 2) [Client]

- 설명: 저장된 코스를 열고, 코스 선과 내 위치를 보며 따라갈 수 있게 한다.
- AC:
  - 코스 Polyline과 내 위치가 동시에 표시된다
  - 진행률(%) 또는 남은 거리 중 최소 1개를 표시한다
  - 이탈(off-route) 감지(거리 임계값)와 복귀 전환(히스테리시스)이 동작한다
  - 턴바이턴(교차로 지시/음성/재탐색)은 제공하지 않는다

### FR-011 코스 메타데이터 v1 자동 산출 [Server]

- 설명: 코스 geometry + POI DB를 기반으로 추천/요약에 필요한 파생치를 계산한다.
- AC:
  - 코스 생성 시 distance/bbox/loop 같은 경량 파생치는 즉시 산출한다
  - amenitiesSummary(코스 주변 버퍼 기반 POI 요약)는 비동기 산출이 가능하다(워커/배치)
  - 파생치 계산 실패는 코스 조회 자체를 막지 않는다(부분 필드 누락 허용)

### FR-012 기본 제공(피처드) 코스 노출 [Server]

- 설명: 앱이 비어있지 않도록 큐레이션 코스를 리스트로 제공한다.
- AC:
  - 지역/노출순서(featuredRank) 기준으로 기본 제공 코스를 조회할 수 있다
  - 피처드 코스는 운영 검증 상태(verifiedStatus)를 구분해 표기할 수 있다

## 2. 비기능 요구사항 (NFR)

### NFR-001 성능

- 주변 조회 p95 < 500ms (단일 인스턴스/로컬 기준)
- 경로 주변 조회 p95 < 800ms (초기 목표)
- 동기화는 페이지 단위 배치 처리(hibernate batch)

### NFR-002 안정성/복구

- 외부 API 호출 실패 시 제한된 재시도(지수 백오프)
- 동기화 실패가 애플리케이션 기동 전체를 막지 않도록 옵션 제공

### NFR-003 보안

- 비밀번호는 반드시 해시(BCrypt)로 저장
- 인증 도입 전에도 민감 데이터(정확한 위치/키)는 로그에 남기지 않는다

### NFR-004 개인정보/데이터 보관

- 위치 이력 보관 기간(예: 30일)을 정책으로 정의
- 데이터 삭제/마스킹 요구 대응 가능해야 함

### NFR-005 관측성(로그/메트릭)

- 동기화/조회 주요 액션 로그에 requestId 또는 핵심 식별자 포함
- 예외는 원인 스택트레이스 포함(서버 내부 로그)

### NFR-006 문서화

- Swagger(OpenAPI)로 모든 엔드포인트 문서화
- 주요 정책/에러코드/권한은 `설계/04_정책정의서.md`에 기록

### NFR-007 배터리/위치 수집

- 코스 따라가기(레벨 2) 시 위치 업데이트 주기/정확도는 배터리와 균형을 맞춘다
- 백그라운드 위치는 MVP에서는 최소화(필요 시 정책 문서로 별도 고지)

### NFR-008 지도/경로 렌더링 성능

- 코스 좌표가 길어도 앱이 끊기지 않도록 단순화/타일링/구간 렌더링 전략을 둔다
- 경로/POI 조회는 캐시와 인덱스로 방어한다
