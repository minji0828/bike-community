## 3. API 계약/정합성 리뷰

### 3.1 (P0) POI 화장실 좌표 필드명 불일치
- Backend DTO: `ToiletResponseDto`는 `lat`, `lon`
- Mobile 타입/사용처: `Toilet`은 `latitude`, `longitude`이며 Map marker에서 그대로 사용

**영향:**
- 런타임에서 `t.latitude`, `t.longitude`가 `undefined`가 되어 지도 마커 표시가 깨질 수 있음

**권장 조치(안전한 방향):**
- 서버 계약(문서/DTO)은 `{lat, lon}`로 유지하고,
- 모바일에서 수신 데이터를 `latitude/longitude`로 **normalize** 하거나, 타입 자체를 `{lat, lon}`로 변경

### 3.2 Geo 네이밍 혼재(기술부채)
- POI/Riding/Course: `lat/lon`
- Location API: `latitude/longitude` + 일부 snake_case

**권장:**
- 단기: 예외를 문서에 명시
- 중기: `/api/v2`에서 Geo 필드 네이밍을 통일

### 3.3 에러 계약 일관성 문제
- `LocationController`는 `ResponseEntity.badRequest()`를 반환하면서도 `ApiResponse.error()`로 body `code=500`을 내보내는 케이스가 존재
- Spring Security가 내보내는 401/403의 body가 `ApiResponse` 형태가 아닐 수 있음(EntryPoint/DeniedHandler 미구현)

**권장:**
- 에러 응답의 `code`는 HTTP status와 일치하도록 통일
- 인증 실패/권한 실패도 `ApiResponse`로 감싸도록 Security 핸들러 추가

### 3.4 스펙 문서 보완 필요(대표)
- Comment cursor/limit 의미(예: `cursor`가 commentId 기반인지, 방향/정렬, max limit 등)
- Meetup list의 status 파라미터(코드가 `all`도 허용한다면 문서에 명시)
- Course request/response의 실제 필드(설계 문서 예시가 일부 단순화되어 있음)

## 4. 코드 품질/보안 리뷰(핵심)

### 4.1 CRITICAL (즉시 수정 권장)

#### A) Location API가 공개되어 타 사용자 위치 조회/변조 가능
- 관련 파일:
  - `backend/src/main/java/com/bikeoasis/global/config/SecurityConfig.java` (permitAll)
  - `backend/src/main/java/com/bikeoasis/domain/user/controller/LocationController.java`

**권장 조치:**
- `/api/v1/locations/**`는 기본적으로 authenticated
- `userId`는 path param을 제거/축소하고 JWT subject(uid)에서 추출
- “관리자용” 엔드포인트는 `X-Admin-Key` 또는 ROLE 기반으로 별도 보호

#### B) POI sync(full refresh 포함) 엔드포인트가 공개됨
- `PoiController`의 `/sync/full`은 DB `deleteAllInBatch()` 수행

**권장 조치:**
- 동기화 API는 admin 전용으로 이동/보호
- 모바일 Settings의 동기화 버튼은 DEV 전용으로 가드하거나 제거

#### C) Course shareId 발급이 인증 없이 가능 + PRIVATE를 UNLISTED로 자동 변경
- `CourseController: POST /{courseId}/share`
- `CourseService.issueShareId()`가 PRIVATE를 UNLISTED로 바꿈

**권장 조치:**
- share 발급은 authenticated + owner 검증
- PRIVATE 자동 전환 제거(정책 문서 기반으로 명시적 동작만 허용)

### 4.2 HIGH (가능한 빨리)

#### A) 외부 API Key 로그 유출
- `SeoulApiClient`가 호출 URL 전체를 로깅 → key 노출

**권장:** key 마스킹 또는 URL 로깅 제거

#### B) JPQL에 `LIMIT` 사용(런타임 실패 위험)
- `UserLocationRepository.findLatestLocationByUserId()`의 JPQL에 LIMIT 사용

**권장:** derived query(`findTopBy...OrderBy...Desc`) 또는 pageable/nativeQuery로 수정

#### C) RestTemplate timeout/retry 설정이 실제 적용되지 않음
- `application.yml`에 timeout 값이 있으나, `RestTemplate`에 connect/read timeout 적용이 보이지 않음

**권장:** request factory로 timeout 적용 또는 `RestClient/WebClient` 전환

#### D) `.env` 실값 커밋
- `frontend/docs/.env`

**권장:** 삭제 + `.env.example`로 대체

### 4.3 MEDIUM (정리하면 안정성이 좋아지는 것)
- 컨트롤러에서 try/catch로 에러 처리 분산 + 내부 메시지 노출 가능 → `GlobalExceptionHandler` 중심으로 정리
- Location의 `isCurrent` 1개 보장 로직이 동시성에 취약(경합 시 multiple current 가능)
- 인메모리 rate limit(무제한 성장/스케일아웃 무력화) → Caffeine/Redis 권장
- startup에서 외부 의존 작업이 기본 활성화(POI fetch, course seed) → profile 분리 권장

### 4.4 LOW (기술부채/정돈)
- `.gitattributes` 범위가 좁아 대규모 EOL 변경이 재발 가능
- 도메인 간 DTO 결합(POI가 `RidingCreateRequest.PointDto` 사용)
- SLF4J 포맷 문자열 오류(`{:.2f}`)
